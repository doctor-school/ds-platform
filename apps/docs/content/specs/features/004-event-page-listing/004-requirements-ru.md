# 004 — Публичная страница мероприятия и листинг ближайших эфиров (Требования)

> Эпик: [«Вебинары» — продуктовый бриф](../../product/webinars/brief-ru.md) · Волна 1, утверждённый вариант A «тонкая вертикаль». Источник PRD: [`004-product-ru.md`](./004-product-ru.md) (US-1…US-6). Источник точности мокапа: завендоренные канвасы в [`design-source/`](../../../../../../design-source/README.md) — `webinar-page.dc.html`, `webinars-listing.dc.html`, `webinar-card.dc.html`.

## Итоги (Outcomes)

- Врач, открывший **распространённую спонсором прямую ссылку**, попадает на полную, серверно отрендеренную страницу вебинара **без аутентификации** — без мягкой стены и без закрытых секций, — и эта ссылка стабильна и рендерится одинаково для гостя и для залогиненного принципала (US-1, US-5).
- Страница мероприятия несёт всё, что нужно врачу, чтобы решить участвовать меньше чем за минуту: название, школа/серия, дата + время с явной пометкой **МСК**, описание, спикеры, скачиваемая **программа PDF**, целевые специальности и партнёры-спонсоры (US-2).
- Страница предлагает **один** основной CTA «Участвовать», ведущий в регистрацию (фича 005) с передачей контекста мероприятия; гость проходит через auth (фича 003), не теряя этого контекста (US-3).
- Минимальный **листинг ближайших эфиров** в приложении `portal` показывает опубликованные мероприятия с датой эфира в будущем, **ближайшими первыми**; каждая карточка несёт достаточно для выбора и ведёт на страницу мероприятия (US-4).
- Страница и листинг **правдиво отражают жизненный цикл** — предстоит / в эфире / завершилось — из **единой state machine** (`draft → published → live → ended → archived`), никогда не устаревший и не противоречивый сигнал; черновики и архивированные мероприятия отсутствуют на публичных поверхностях, а ранее распространённая ссылка на **архивированное** мероприятие рендерит публичную плашку «мероприятие в архиве» **без** мёртвого CTA (US-6).
- Read-поверхность собрана из токенов `@ds/design-system` по геометрии завендоренного канваса (нео-брутализм), проверена на обоих брейкпоинтах × обеих темах (ADR-0013).

## Scope

**В scope:**

- **Публичная страница мероприятия** (`portal`, серверный рендер) для мероприятия в состоянии `published` / `live` / `ended` — публично читаема, без auth, полный контент-набор US-2, построена по `webinar-page.dc.html`.
- **Рендер жизненного цикла** на странице — переключение `status` `upcoming | live | ended` (hero-бейдж, плата времени, пара CTA, нижний CTA), соответствующее enum-пропу `status` канваса.
- **Обработка прямой ссылки на архивированное мероприятие** — публичная плашка «мероприятие в архиве» без CTA участия (решение владельца, вариант «а»).
- **Единственный CTA «Участвовать»** на странице → ведёт во флоу регистрации, передавая контекст мероприятия (механика регистрации + передача гость→auth принадлежат 005/003 — см. Вне scope).
- **Листинг ближайших эфиров** (`portal`) — мероприятия `published` ∧ дата эфира в будущем, отсортированные ближайшей датой первыми, **с группировкой по дням** по ритму §09, построен по `webinars-listing.dc.html` (список карточек с группировкой по дням), рендерящий юнит `webinar-card.dc.html`; включает **пустое состояние** канваса.
- **Публичный read-model + query-эндпоинты** в `apps/api` — publish-safe проекция `PublicEventPage` и списочная проекция `UpcomingBroadcastCard`, обе классифицированы **public** в матрице endpoint-authz (ADR-0001).
- **Каноничное представление московского времени** — read-model хранит каноничный момент; каждая поверхность представляет его в `Europe/Moscow` с явной пометкой **МСК**.
- **RU-only копирайт из типизированного каталога сообщений** — без хардкод-строк, переиспользуя i18n-готовую структуру, заложенную в 003 (EARS-21).
- **Seed/фикстур-мероприятия**, достаточные для рендера и E2E-прогона двух поверхностей до того, как фича 007 привнесёт авторство в админке (отслеживаемый шов — см. Зависимости).

**Явно вне scope** (каждое — именованная отсрочка, не молчаливый дефолт):

- **Авторство мероприятий и переходы жизненного цикла** — создание/редактирование мероприятия, настройка стрима и прогон `draft → published → live → ended → archived` принадлежат **фиче 007** (admin/Refine). 004 потребляет итоговое состояние только на чтение и верифицируется на seed-мероприятиях, пока 007 не приземлится (Зависимости).
- **Механика регистрации** — регистрация в один тап для залогиненного, гость→auth→завершение, видимость «мои события» и состояние регистрации на странице мероприятия принадлежат **фиче 005**. 004 владеет только CTA и передачей контекста.
- **Комната эфира** (встроенный плеер, живой чат, heartbeat-присутствие) за путём присоединения к эфиру — **фича 006**. CTA состояния «в эфире» у 004 ведёт к ней; сама комната и её серверное гейтирование — 006.
- **Календарь / месячный вид, фасеты по специальностям, фильтр «Только с НМО», недельная пагинация и полнотекстовый поиск** — волна 2. Завендоренные `webinars-listing.dc.html` и `webinars-month.dc.html` показывают эти контролы; волна 1 отгружает только минимальный список с группировкой по дням и сортировкой ближайшими первыми без них. (`webinars-month.dc.html` в 004 не строится.)
- **Архив записей, отзывы, рейтинги спикеров, фото/видео-отчёты** — волна 2+.
- **Конгрессы, офлайн-мероприятия, платные тарифы** — вне эпика.

## Ограничения (Constraints)

- **Публичная читаемость — серверная, а не клиентская мягкая стена.** Страница мероприятия для `published` / `live` / `ended` рендерится серверно и полностью читаема с нулевой аутентификацией; серверно закрыта только **комната** за путём присоединения (фича 006). Легаси-оверлей «авторизуйтесь для просмотра» для анонима — запрещённый паттерн (recon §7a, §8).
- **Единая state machine.** Публичная видимость и рендерящееся состояние выводятся из одного enum `EventLifecycleState` (`draft → published → live → ended → archived`) — никогда не из россыпи булевых флагов (recon §7d). И страница, и карточка читают одно состояние; они не могут разойтись.
- **Никакого хардкода origin / URL.** Публичный URL мероприятия, origin портала и эндпоинты API читаются из конфигурации, никогда не хардкодятся в коде или спеке (AGENTS.md §9).
- **Каноничный часовой пояс.** Время хранится как каноничный момент и представляется в `Europe/Moscow` с пометкой **МСК**; представление не должно съезжать в локальный часовой пояс зрителя (recon §7d — легаси-пробел «всегда Москва, без TZ» закрывается тем, что МСК становится явным, а не подразумеваемым).
- **Канвас — спецификация точности (ADR-0013 / ADR-0014 §4).** Страница мероприятия и карточка строятся из токенов `@ds/design-system` по геометрии завендоренного `.dc.html` (границы 2px, тень `6px 6px 0`, плата времени, десктоп-грид `196px 1fr`, плоский full-bleed ≤900px). Произвольные значения Tailwind заблокированы линтером; где проза и канвас расходятся, побеждает канвас.
- **Стек** (ADR-0002 / ADR-0004): query-эндпоинты — NestJS + `nestjs-zod`, SSOT схем в `packages/schemas/`, URI-версионирование `/v1/...`, Vitest + supertest; портал — Next.js 15 с серверно отрендеренными публичными маршрутами. Зависящие от сервиса тесты `skipIf` при отсутствии их env-зависимости, чтобы не краснить общий CI unit job.
- **Publish-safe проекция.** Публичный read-model отдаёт только publish-safe поля мероприятия — никаких операторских заметок, коммерческих условий спонсора и PII зарегистрированных (recon §6 — легаси-ростер `getEmailsForOrder` никогда не на публичной поверхности).

## Модель событий (Event Model)

Фича 004 — **read-сторона** агрегата вебинара. Создание мероприятий и переходы жизненного цикла — команды, принадлежащие **фиче 007**; 004 владеет публичными **запросами**, publish-safe **read-моделями** и **политикой видимости** над единым состоянием жизненного цикла. Она не эмитит событий жизненного цикла — она читает состояние, которое те оставляют.

### Запросы (обрабатываются `apps/api`, оба классифицированы **public** — без auth)

`GetPublicEventPage(idOrSlug)` · `ListUpcomingBroadcasts()`

### События (потребляются только на чтение; принадлежат переходам фичи 007)

| Событие          | Владелец (007) | Что читает 004                                                                         |
| ---------------- | -------------- | -------------------------------------------------------------------------------------- |
| `EventPublished` | 007            | Мероприятие становится публично достижимым и попадает в кандидаты листинга ближайших.  |
| `EventWentLive`  | 007            | Страница + карточка рендерят сигнал «в эфире»; CTA ведёт к комнате.                    |
| `EventEnded`     | 007            | Страница рендерит состояние «завершён»; карточка выпадает из листинга ближайших.       |
| `EventArchived`  | 007            | Мероприятие покидает все публичные листинги; прямая ссылка рендерит плашку «в архиве». |

### Read-модели

- **`PublicEventPage`** — publish-safe проекция одного мероприятия: `id`, `slug`, `title`, `school` (серия), `startsAt` (каноничный момент), `durationMin`, `description`, `speakers[]` (имя + регалии, без контактной PII), `programPdfUrl?`, `specialties[]`, `partners[]`, `state` (`published | live | ended | archived`). У черновиков нет публичной проекции.
- **`UpcomingBroadcastCard`** — проекция карточки листинга: `id`, `slug`, `title`, `school`, `startsAt`, `specialties[]`, `speakers[]` (имена), `state` (`published | live`). Сортировка по `startsAt` по возрастанию.
- **`EventLifecycleState`** — единый enum `draft | published | live | ended | archived`, который читают обе поверхности.

### Политики

- **Видимость публичной страницы** — `published`, `live` и `ended` рендерят полную публичную страницу; `draft` **публично недостижим** (not-found); `archived` рендерит **плашку «в архиве»** (без CTA). Страница остаётся открытой через `live` и `ended`; закрыт только путь присоединения к комнате (006).
- **Членство в листинге ближайших** — карточка появляется **тогда и только тогда**, когда мероприятие `published` (или `live`) **и** его дата эфира в будущем / сейчас в эфире; `draft`, `ended` и `archived` не появляются никогда.
- **Кросс-поверхностная согласованность** — карточка и страница выводят `state` из одного `EventLifecycleState`; мероприятие «в эфире» читается как «live» на обеих поверхностях, так что они не могут показать противоречивый сигнал.

## EARS-требования

> **Соглашение о нумерации:** плоское (`EARS-1`, `EARS-2`, …) по ADR-0006 §4. EARS-1…9 — функциональные обработчики (каждый становится дочерним Issue); EARS-10…14 — кросс-срезовые повсеместные / нежелательного-поведения требования, применяемые по всей read-поверхности. Каждое положение несёт бэклинк `realizes: US-N` к истории PRD, которую формализует (ADR-0014 §2).

**Публичная страница мероприятия**

- **EARS-1** _(realizes: US-1, US-5)_ — Когда посетитель запрашивает мероприятие в состоянии `published`, `live` или `ended` по его стабильному публичному URL, система должна отрендерить полную страницу мероприятия **серверно, не требуя аутентификации**, возвращая байт-в-байт одинаковый публичный контент для гостя и для залогиненного принципала (распространённая спонсором ссылка «просто работает» для любого получателя).
- **EARS-2** _(realizes: US-2)_ — Страница мероприятия должна представить полный набор для решения из read-модели `PublicEventPage` — название, кикер школы/серии, **дату и время старта с явной пометкой МСК**, описание, спикеров (имя + регалии), чипы целевых специальностей, партнёров-спонсоров и, когда у мероприятия есть, скачиваемую ссылку на **программу PDF** — свёрстанные по `webinar-page.dc.html`.
- **EARS-3** _(realizes: US-3)_ — Страница мероприятия должна представить ровно **один** основной CTA «Участвовать», ведущий посетителя во флоу регистрации (фича 005) с передачей контекста мероприятия; гость проходит через auth (фича 003) и возвращается завершить регистрацию, **не теряя** этого контекста. (004 владеет только CTA и передачей контекста; механика регистрации и раунд-трип auth принадлежат 005/003.)
- **EARS-4** _(realizes: US-6)_ — Страница мероприятия должна отражать текущее состояние жизненного цикла из единого `EventLifecycleState` — **предстоит** (CTA регистрации), **в эфире** (сигнал «идёт сейчас» + CTA, ведущий к комнате, фича 006) или **завершилось** (состояние «запись/завершён» **без мёртвого CTA**) — переключая hero-бейдж, плату времени и аффорданс CTA согласно enum `status` канваса и никогда не показывая сигнал, противоречащий машине.

**Видимость и архивные ссылки**

- **EARS-5** _(realizes: US-6, US-5)_ — Когда посетитель запрашивает страницу **архивированного** мероприятия по ранее распространённой прямой ссылке, система должна отрендерить **публичную плашку «мероприятие в архиве» без CTA участия** (решение владельца, вариант «а») — никогда `404`, редирект на листинг или живой CTA. (Закрывает открытый вопрос PRD о поведении архивной ссылки; распространённая ссылка деградирует мягко, а не в тупик.)
- **EARS-6** _(realizes: US-6)_ — **Черновик** мероприятия не должен быть публично достижим — запрос страницы черновика возвращает not-found (неотличимо от несуществующего мероприятия), и ни один черновик никогда не появляется в публичном листинге. Публично рендерятся только `published` / `live` / `ended`; `archived` — это плашка EARS-5.

**Листинг ближайших эфиров**

- **EARS-7** _(realizes: US-4)_ — Портал должен представить **листинг ближайших эфиров** из мероприятий, которые `published` или `live` и чья дата эфира в будущем или сейчас в эфире, отсортированный **ближайшей датой эфира первой**, с группировкой по дням по ритму §09, построенный по `webinars-listing.dc.html` (волна 1 отгружает минимальный список — **без** фасетов по специальностям, недельной пагинации, месячного вида и поиска).
- **EARS-8** _(realizes: US-4)_ — Каждая карточка листинга должна нести набор для выбора из проекции `UpcomingBroadcastCard` — **дату + время старта (МСК)**, название, школу/серию, целевые специальности и спикеров — отрендеренные как юнит `webinar-card.dc.html`, и должна вести на страницу этого мероприятия (EARS-1).
- **EARS-9** _(realizes: US-6, US-4)_ — Когда мероприятие **в эфире**, и его карточка листинга, и его страница мероприятия должны показывать сигнал «идёт сейчас», выведенный из одного `EventLifecycleState`, так что врач никогда не видит противоречивого состояния между двумя поверхностями; карточка, чьё мероприятие `ended` или `archived`, должна выпасть из листинга при следующем чтении.

**Кросс-срезовое (повсеместное / нежелательное поведение)**

- **EARS-10** _(realizes: US-1, US-5)_ — Эндпоинты `GetPublicEventPage` и `ListUpcomingBroadcasts` должны быть классифицированы **public** в матрице endpoint-authz (ADR-0001 §2), не требовать **никакой** аутентификации и отдавать **только** publish-safe проекцию — никогда тело мероприятия `draft`/`archived`-как-активного, операторские/коммерческие поля или PII зарегистрированных.
- **EARS-11** _(realizes: US-4)_ — Когда не существует ни одного мероприятия `published`/`live` с датой в будущем, листинг должен отрендерить **пустое состояние** канваса (ясный аффорданс «нет ближайших эфиров»), а не пустую или сломанную поверхность.
- **EARS-12** _(realizes: US-2, US-6)_ — Каждая дата/время на обеих поверхностях должны быть представлены в **`Europe/Moscow` с явной пометкой МСК**, вычислены из каноничного момента read-модели, и не должны съезжать в локальный часовой пояс зрителя.
- **EARS-13** _(realizes: US-1, US-2, US-3, US-4)_ — Поверхности вебинаров в портале должны рендериться на **русском** с **отсутствием хардкод-строк** — весь копирайт (лейблы, статус-бейджи, текст CTA, копирайт плашки «в архиве», копирайт пустого состояния) берётся из типизированного каталога сообщений, заложенного в 003 (EARS-21), поверх i18n-готовой структуры, чтобы позднюю локаль можно было добавить без правки компонентов.
- **EARS-14** _(realizes: US-1, US-2, US-4)_ — Страница мероприятия и карточка листинга должны быть построены из **токенов `@ds/design-system`** по геометрии завендоренного канваса (`webinar-page.dc.html`, `webinar-card.dc.html`, `webinars-listing.dc.html`) — границы 2px, тень `6px 6px 0`, десктоп-грид `196px 1fr`, плоский full-bleed ≤900px, плата времени — корректно рендерясь на **обоих брейкпоинтах × обеих темах**; произвольные значения Tailwind заблокированы линтером (ADR-0013).

## Инварианты

- Ни одно непубличное мероприятие не выставляется на публичную поверхность: страница `draft` — not-found, страница `archived` — плашка EARS-5, и ни `draft`, ни `ended`, ни `archived` не появляются в листинге ближайших (EARS-6, EARS-7, EARS-10).
- Публичная страница мероприятия и карточка листинга выводят рендерящееся состояние жизненного цикла из **одного** `EventLifecycleState`; две поверхности никогда не могут показать противоречивое состояние для одного мероприятия (EARS-9).
- Публичный read-model и эндпоинты отдают только publish-safe поля — никаких операторских/коммерческих данных и PII зарегистрированных (EARS-10).
- Каждая дата/время, отрендеренные на любой поверхности, — в `Europe/Moscow` с пометкой МСК, выведены из одного каноничного момента (EARS-12).
- Публичная страница мероприятия полностью читаема с нулевой аутентификацией в `published` / `live` / `ended`; рендеры для гостя и аутентифицированного контент-идентичны (EARS-1). Закрыт только путь присоединения к комнате (фича 006), никогда не страница.
- Страница мероприятия несёт ровно один основной CTA участия; в состоянии `ended` этот CTA отсутствует, а не является мёртвой ссылкой (EARS-3, EARS-4).

## Зависимости и секвенирование

- **Event read-model + seed (отслеживаемый шов → фича 007).** 004 владеет read-моделью `PublicEventPage` / `UpcomingBroadcastCard` и двумя публичными query-эндпоинтами, но **авторство мероприятий и переходы `draft → published → live → ended → archived` принадлежат фиче 007** (admin/Refine). Пока 007 не приземлится, 004 строится и E2E-прогоняется на **seed-фикстур-мероприятиях**, покрывающих каждое состояние жизненного цикла. Это отслеживаемый шов, а не молчаливый стаб: критерий «сделано против реальной зависимости» — _«поверхности 004 рендерят мероприятия, авторизованные и переведённые через 007, а не только seed-фикстуры»_, несомый на блокирующей связке 004↔007, которую вяжет `open-ears-issues` (его шаг 4).
- **Передача регистрация/auth (фича 005 / 003).** CTA «Участвовать» из EARS-3 передаёт контекст мероприятия во флоу регистрации. Auth (003) уже отгружен; механика регистрации (005) — **более поздняя вертикаль**. E2E 004 стабает цель передачи и проверяет только, что CTA входит во флоу с сохранённым контекстом мероприятия — завершённая регистрация верифицируется в 005.
- **Комната эфира (фича 006).** CTA состояния live из EARS-4/EARS-9 ведёт _к_ комнате; комната и её серверное гейтирование присоединения — 006. 004 проверяет цель ведения, а не комнату.
- **Дизайн-система (`@ds/design-system`, ADR-0013).** Нео-брутал-токены/примитивы, на которых строятся страница и карточка, отгружаются из DS-фундамента (#510 → #512–515, на `main`). Юнит webinar-card (`webinar-card.dc.html`) уже завендорен (вытянут рано для ритм-демо #514). Не ожидается, что 004 выпустит новый DS-примитив; любой bespoke-элемент, который канвас требует сверх инвентаря DS, сначала прогоняет гейт `build-ui-from-design-system` и фиксируется в PR.
- **Dev-стенд Postgres.** e2e-тесты read-модели гоняются против Postgres дев-стенда (`DATABASE_URL` из `.env.local`); E2E портала гоняется против живого стенда по `.claude/rules/dev-stand.md`.
