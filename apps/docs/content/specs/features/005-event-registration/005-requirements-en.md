# 005 — Event registration & «мои события» (Requirements)

> Epic: [Webinars — product brief](../../product/webinars/brief.md) · Wave 1, approved variant A «thin vertical». PRD source: [`005-product.md`](./005-product.md) (US-1…US-5). Mockup source of truth: the vendored canvases in [`design-source/`](../../../../../../design-source/README.md) — `my-events.dc.html`, `webinar-page.dc.html` (registered state), `webinar-card.dc.html` (registered variant).

## Outcomes

- A **logged-in doctor** registers for a webinar in **one action** on the event page, and the page reflects the registered state immediately — committing takes seconds (US-1, US-3).
- A **guest** who taps «Участвовать» is carried through the 003 login/signup flow and comes out **registered for that same event**, landing back on that event page in the registered state — never dropped back to re-find the event (US-2). This retires the legacy "postponed registration" parking mechanism.
- The event page shows the doctor's **true registration state** — a registered doctor sees a confirmation + join signposting, not the register CTA again; an unregistered doctor sees the register CTA (US-3).
- Every registered event is visible in a personal **«мои события»** surface in the portal account, nearest first, with date/time **МСК** and a link back to each event page — closing the legacy "I registered but can't find it" gap; a just-registered event appears there immediately (US-4).
- Every registration is **durably recorded server-side against the authenticated account** — one doctor + one event = at most one registration, regardless of path or repeats — and is the basis for room admission (feature 006) and the trustworthy sponsor roster (US-5).
- The registered surfaces are built from `@ds/design-system` tokens to the vendored canvas geometry (neo-brutalist language), verified at both breakpoints × both themes (ADR-0013).

## Scope

**In:**

- **Logged-in one-tap registration** — an authenticated doctor activates «Участвовать» on a `published`/`live` event page and a registration is recorded against their account in one action (EARS-1), the page swapping to the registered state without a second step.
- **Guest-through-auth completion** — a guest activating «Участвовать» is taken through the 003 login/signup flow carrying the **event context**, and on auth success the registration for that same event completes and the doctor lands back on that event page registered (EARS-2). Replaces the retired legacy "postponed registration" mechanism.
- **The one-registration invariant** — one doctor + one event = at most one registration, enforced server-side regardless of how many times or through which path the doctor registers; a repeat is an idempotent no-op returning the existing registration (EARS-3).
- **Registration state on the event page** — the page reflects the authenticated doctor's true registration state (registered → confirmation + join signposting; unregistered → the 004 register CTA), read per-user **without** contaminating 004's public, cacheable page projection (EARS-4).
- **Join signposting** — for a registered doctor, the `upcoming` state shows when/how they will join; the `live` state makes the path onward to the room (feature 006) obvious (EARS-5).
- **«Мои события» account surface** (`portal`, authenticated) — the doctor's registered upcoming events, nearest first, each with date/time **МСК** and a link back to its event page, built to `my-events.dc.html` (the **Предстоящие** tab; Записи / Сертификаты tabs are out — see below) (EARS-6), a just-registered event appearing immediately (EARS-7).
- **Durable registration record + per-user read models** in `apps/api` — the registration write, and the `EventRegistrationState` (per-caller, per-event) and `MyEvents` (per-caller list) read models, all **`doctor_guest`-authenticated** in the endpoint-authz matrix (ADR-0001 §2). The record is the basis for room admission (006) and the sponsor roster (EARS-8, EARS-10).
- **Registration lifecycle gating** — registration is offered while the event is `published` and upcoming, or `live` (register-during-live is a normal path), and **not** offered for `ended`/`archived` (EARS-9). No per-event configurable cutoff beyond event state (owner decision 2026-07-06).
- **Canonical Moscow-time presentation** on «мои события» — every date/time in `Europe/Moscow` explicitly labeled **МСК**, computed from the read model's canonical instant (EARS-11).
- **RU-only copy from the typed message catalog** — no hardcoded user-facing strings, reusing the i18n-ready structure established in 003 (EARS-21) and 004 (EARS-13) (EARS-12).

**Explicitly out** (each a named deferral, not a silent default):

- **Registration cancellation** — a doctor **cannot** cancel a registration in wave 1 (owner decision 2026-07-06). Every recorded registration is current; there is **no** cancelled state in the data model or the roster, and the surfaces present no cancel affordance. Deferred to **wave 2 with the notifications work** (cancellation + its confirmation/reminder emails ship together).
- **Welcome / confirmation emails and reminders** — wave 2 (notifications). Registration records the doctor server-side; it sends no email in wave 1.
- **Room access enforcement** — admitting exactly the registered audience into the live room is owned by **feature 006**. 005 records the registration that 006 admits against; the `live`-state signposting routes _toward_ the room, but the server-side join gating is 006.
- **Event authoring & lifecycle transitions** — `draft → published → live → ended → archived` and the event content are owned by **feature 007** (admin/Refine). 005 reads the lifecycle state read-only to gate the register affordance and is verified against seeded events until 007 lands (Dependencies).
- **The public event page + upcoming listing** — the page content, the single «Участвовать» CTA, and the listing are owned by **feature 004**. 005 owns only the per-user registration state overlaid on that page and the «мои события» surface; it consumes 004's CTA and event-context handoff.
- **The sponsor roster export / attendance report** — 005 owns the durable registration record (the roster's membership basis); the report generation + presence minutes are the epic's later verticals (006 heartbeat + wave-2 «Отчёт партнёра»).
- **«Мои события» Записи / Сертификаты tabs, specialty filter** — the `my-events.dc.html` canvas shows tabs Предстоящие / Записи / Сертификаты and a specialty filter; wave 1 ships **only** the **Предстоящие** tab (registered upcoming events). Recordings archive and certificates are wave 2+.
- **Registration limits, closed/whitelist events, promo codes, paid registration** — out of the epic / disabled in legacy.

## Constraints

- **Auth is the prerequisite, reused not reimplemented.** Registration is a `doctor_guest`-authenticated action; the guest path routes through the **already-shipped 003** login/signup flow and reuses its BFF session (003 EARS-8). 005 adds **no** auth primitive — no new credential, session, or token path (ADR-0001; 003 Constraints).
- **Event context survives the auth round-trip.** The guest-through-auth flow carries a **safe** registration-intent — the event slug + a same-origin return target only, **never** PII or a credential — through the 003 round-trip, so the doctor returns to the originally chosen event and completes registration without re-search or a second «Участвовать» tap. A cross-origin or open return target is a banned pattern (open-redirect).
- **Per-user state never contaminates the public page.** 004's `GetPublicEventPage` is **public, content-identical for guest and principal, and cacheable** (004 EARS-1/EARS-10). The doctor's registration state is a **separate authenticated read** (`EventRegistrationState`), composed onto the page for a logged-in caller — it is never folded into the public projection or its shared cache.
- **The one-registration invariant is server-enforced.** At most one registration per (doctor, event) is guaranteed by a database uniqueness constraint, not by client discipline; a repeat register is an idempotent no-op (ADR-0003 §5). No path — one-tap, guest-through-auth, or «мои события» re-entry — can create a duplicate.
- **No cancelled state.** Every registration row is current (owner decision). There is no `cancelled` column, no soft-delete, and no roster distinction for cancelled registrations in wave 1 — cancellation is a wave-2 vertical, not a silent seam.
- **Registration gating derives from the single event state machine.** The register affordance is offered iff the event is `published` (upcoming) or `live`, and withheld for `ended`/`archived`, read from the one `EventLifecycleState` (004 / recon §7d) — never a per-event boolean or a configurable cutoff.
- **No hardcoded origin / URL.** The portal origin, the return target, and API endpoints are read from configuration, never hardcoded in code or spec (AGENTS.md §9).
- **Canonical timezone.** «Мои события» presents each event's canonical instant in `Europe/Moscow` labeled **МСК**; the presentation must not drift to the viewer's local timezone (mirrors 004 EARS-12).
- **Canvas is the fidelity spec (ADR-0013 / ADR-0014 §4).** The registered event-page state, the `registered` card variant, and «мои события» are built from `@ds/design-system` tokens to the vendored `.dc.html` geometry (2px borders, `6px 6px 0` shadow, time plate, day-grouped rhythm). Arbitrary Tailwind values are lint-blocked; where prose and the canvas disagree, the canvas wins.
- **Stack** (ADR-0002 / ADR-0004): the registration command + per-user reads are NestJS + `nestjs-zod`, schema SSOT in `packages/schemas/`, URI versioning `/v1/...`, Vitest + supertest; the portal is Next.js 15. Service-dependent tests `skipIf` their dependency env is absent so they do not redden the shared CI unit job.

## Event Model

Feature 005 is the **write side** of webinar registration plus two **per-user read sides**. It owns the registration command, the durable registration record, and the per-caller read models; it **reads** the event lifecycle state (owned by 007, surfaced publicly by 004) to gate the register affordance, and it **produces** the record that feature 006 admits against and the sponsor roster is drawn from.

### Commands (handled by `apps/api`, `doctor_guest`-authenticated)

`RegisterForEvent(idOrSlug)` — the single registration command; fires on the logged-in one-tap path and on the guest's post-auth return. Idempotent: at most one registration per (doctor, event).

The **guest-through-auth** completion is not a second server command — it is the same `RegisterForEvent` fired after the 003 flow establishes the session, carrying the event context through the round-trip (design §3). There is no server-side "postponed registration" record (the retired legacy mechanism).

### Events

| Event                                                               | Owner                    | Notes                                                                                                                                                                       |
| ------------------------------------------------------------------- | ------------------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `DoctorRegisteredForEvent`                                          | `apps/api`               | Durable registration recorded (idempotent). Basis for room admission (006) + the sponsor roster; one terminal `audit_ledger` row. A repeat register emits **no** new event. |
| `EventPublished` / `EventWentLive` / `EventEnded` / `EventArchived` | 007 (consumed read-only) | 005 reads the resulting `EventLifecycleState` to gate the register affordance (offered for `published`/`live`, withheld for `ended`/`archived`).                            |

### Read models

- **`EventRegistrationState`** — per authenticated caller, per event: `{ registered: boolean, registeredAt? }`. An **authenticated** read, composed onto the 004 event page for a logged-in doctor; never part of the public `PublicEventPage` projection or its shared cache.
- **`MyEvents`** — the authenticated doctor's registered **upcoming** events, ordered nearest `startsAt` first: each item `{ eventId, slug, title, school, startsAt (canonical instant), state ∈ {published, live} }`. Feeds the «мои события» **Предстоящие** tab.
- **`EventRoster`** — the set of current registrations for an event (owned here; **consumed** by feature 006 for admission and by the wave-2 sponsor report). Never exposed on a public surface — no registrant PII leaks to 004's public endpoints (recon §6).

### Policies

- **On `RegisterForEvent`** for a `published` (upcoming) or `live` event → upsert **at most one** registration for `(user_id, event_id)` (a repeat is a no-op returning the existing registration), append one `audit_ledger` row on first insert, and reflect the registered state on the event page + «мои события». For an `ended`/`archived` event → the affordance is absent and the command is refused (EARS-9).
- **Registration is durable and current** — no cancelled state; a recorded registration remains until (wave 2) cancellation exists.
- **Guest path** → the event context is captured and carried through the 003 auth flow; on session establishment the same policy runs (EARS-2).

## EARS requirements

> **Numbering convention:** flat (`EARS-1`, `EARS-2`, …) per ADR-0006 §4. EARS-1…9 are the functional handlers (each becomes a child Issue); EARS-10…13 are cross-cutting ubiquitous / unwanted-behavior requirements enforced across the surface. Each clause carries a `realizes: US-N` backlink to the PRD story it formalizes (ADR-0014 §2).

**Registration (logged-in + guest)**

- **EARS-1** _(realizes: US-1, US-3)_ — When an **authenticated** doctor activates «Участвовать» on an event whose state is `published` (upcoming) or `live`, the system shall record a registration against their account in **one action** and reflect the registered state on the event page immediately — no second step, no confirmation round-trip.
- **EARS-2** _(realizes: US-2)_ — When a **guest** activates «Участвовать», the system shall carry the **event context** into the 003 login/signup flow and, on auth success, complete the registration for **that same event**, returning the doctor to that event page in the registered state — the doctor never has to find the event again, re-search, or tap «Участвовать» a second time. The event context is a safe, same-origin registration-intent (event slug + return target only — never PII or a credential); a cross-origin return target is rejected.
- **EARS-3** _(realizes: US-1, US-5)_ — One doctor + one event shall yield **at most one** registration, regardless of how many times or through which path (one-tap, guest-through-auth, «мои события» re-entry) the doctor registers; a repeat registration is an **idempotent no-op** returning the existing registration — the system shall create no duplicate row and emit no second `DoctorRegisteredForEvent`. The invariant is enforced by a database uniqueness constraint on `(user_id, event_id)`, not by client discipline.

**Registration state on the event page**

- **EARS-4** _(realizes: US-3)_ — The event page shall reflect the **authenticated** doctor's true registration state — **registered** → a confirmation + join signposting (EARS-5) replacing the register CTA; **unregistered** → the 004 «Участвовать» register CTA — read per-user via the `EventRegistrationState` model **without** contaminating 004's public, cacheable page projection. A registered doctor is **never** shown the register CTA as if unregistered.
- **EARS-5** _(realizes: US-3)_ — For a registered doctor, the event page shall signpost **how/when they will join**: in the `upcoming` state, when the broadcast starts (date/time МСК) and that they are registered; in the `live` state, an obvious path **onward to the room** (feature 006). The signposting is built to the `webinar-page.dc.html` `upcoming`/`live` registered states and the `webinar-card.dc.html` `registered` variant.

**«Мои события»**

- **EARS-6** _(realizes: US-4)_ — The portal account shall present a **«мои события»** surface (the **Предстоящие** tab of `my-events.dc.html`) listing the authenticated doctor's registered **upcoming** events (`published`/`live`, future or currently airing), ordered **nearest first**, each carrying date/time (МСК), title, school/series, and a link back to that event's page — closing the legacy "registered but can't find it" gap. (Wave 1 ships only the Предстоящие tab — no Записи / Сертификаты tabs, no specialty filter.)
- **EARS-7** _(realizes: US-4)_ — When a doctor registers for an event (via any path), that event shall appear in their «мои события» **immediately** on the next read — a just-registered event is never missing from the list.

**Recording & gating**

- **EARS-8** _(realizes: US-5)_ — Every registration shall be **durably recorded server-side against the authenticated account** and be the single basis for room admission (feature 006) and the sponsor roster — the roster admits/attributes exactly the recorded registrations. The registration record carries no more than the `(doctor, event, registeredAt)` fact; no registrant PII is ever exposed on a public surface (recon §6).
- **EARS-9** _(realizes: US-1, US-2)_ — Registration shall be **offered** while the event is `published` (upcoming) or `live` (register-during-live is a normal path leading straight toward the room), and **not offered** for `ended`/`archived` — the affordance is absent and the `RegisterForEvent` command is refused for those states. Gating derives from the single `EventLifecycleState`; there is **no** per-event configurable cutoff beyond event state (owner decision 2026-07-06). (`draft` is not publicly reachable — 004 EARS-6.)

**Cross-cutting (ubiquitous / unwanted-behavior)**

- **EARS-10** _(realizes: US-1, US-4, US-5)_ — The `RegisterForEvent` command, the `EventRegistrationState` read, and the `MyEvents` read shall be classified **`doctor_guest`-authenticated** in the endpoint-authz matrix (ADR-0001 §2) — an unauthenticated caller is taken through the 003 auth flow (EARS-2), **never** silently succeeds, and never receives another doctor's registration state or roster. The per-user reads return only the caller's own registrations; no endpoint exposes the `EventRoster` publicly or another doctor's data.
- **EARS-11** _(realizes: US-4)_ — Every date/time on «мои события» (and the registered-state signposting) shall be presented in **`Europe/Moscow`, explicitly labeled МСК**, computed from the read model's canonical instant, and shall not drift to the viewer's local timezone (mirrors 004 EARS-12).
- **EARS-12** _(realizes: US-1, US-2, US-3, US-4)_ — The 005 portal surfaces shall render in **Russian** with **no hardcoded user-facing strings** — all copy (the registered-state confirmation, join signposting, the «мои события» labels/tabs/empty-state, CTA text) sourced from the typed message catalog established in 003 (EARS-21) and 004 (EARS-13), over the i18n-ready structure.
- **EARS-13** _(realizes: US-3, US-4)_ — The registered event-page state, the `registered` card variant, and «мои события» shall be built from **`@ds/design-system` tokens** to the vendored canvas geometry (`webinar-page.dc.html`, `webinar-card.dc.html` registered variant, `my-events.dc.html`) — 2px borders, `6px 6px 0` shadow, time plate, day-grouped rhythm — rendering correctly at **both breakpoints × both themes**; arbitrary Tailwind values are lint-blocked (ADR-0013), reusing the `webinar-card` unit built in 004.

## Invariants

- One doctor + one event = **at most one** registration, enforced by the `(user_id, event_id)` uniqueness constraint; no path creates a duplicate and a repeat register is a no-op (EARS-3).
- Every registration row is **current** — there is no cancelled state, no soft-delete, and no roster distinction for a cancelled registration in wave 1 (owner decision; EARS-8, Scope → Out of scope).
- The event page never shows the register CTA to a doctor who is already registered, and never shows the registered state to a doctor who is not (EARS-4).
- A registration is only creatable while the event is `published` (upcoming) or `live`; the command is refused for `ended`/`archived` and impossible for `draft` (not publicly reachable) (EARS-9).
- The doctor's registration state and the `EventRoster` are **never** on a public surface — 004's public page/listing stay content-identical for guest and principal and carry no registrant PII (EARS-4, EARS-8, EARS-10).
- The guest-through-auth flow returns the doctor to the **originally chosen** event, registered — the event context is never lost across the 003 round-trip (EARS-2).
- Every date/time on «мои события» is in `Europe/Moscow` labeled МСК, derived from one canonical instant (EARS-11).
- The registration command emits exactly one terminal `audit_ledger` entry on first insert, and none on an idempotent repeat (EARS-3, EARS-8).

## Dependencies & sequencing

- **Auth foundation (feature 003) — shipped.** The guest-through-auth path reuses 003's login/signup + BFF session (003 EARS-8). 003 is already in dev/shipping; 005 adds no auth primitive and consumes the session as the registering principal. The guest flow's only 005-owned mechanic is the safe event-context carry across the round-trip (design §3).
- **Public event page + CTA (feature 004).** 004 owns the event page, the single «Участвовать» CTA, and the event-context handoff into "the registration flow". 005 lands the actual registration behind that CTA and the per-user registered state overlaid on the page. 005 is sequenced **after** 004 is on `main` (the CTA + handoff exist to build on); until then 005's E2E drives the CTA 004 provides against seeded events. This is the 004↔005 blocking link `open-ears-issues` wires (its step 4).
- **Event read model + seed (tracked seam → feature 007).** Registration gating (EARS-9) reads the `EventLifecycleState` owned by 007. Until 007 lands, 005 is built and E2E-driven against **seeded fixture events** in each state (published/live/ended/archived). The "done against the real dependency" criterion is _"registration gates on events authored + transitioned through 007, not only seeds"_, carried on the 005↔007 blocking link.
- **Webinar room (feature 006).** EARS-5/EARS-8's `live`-state signposting routes _toward_ the room, and the registration record is what 006 admits against. The room and its server-side join gating are 006; 005 asserts the routing target and produces the record, not the admission.
- **Sponsor roster / report.** 005 owns the durable `EventRoster` membership; the roster export + presence-minute report are the epic's later verticals (006 heartbeat + wave-2 report). 005's DoD is "the registration is recorded and admits/attributes correctly", not "the report is generated".
- **Design system (`@ds/design-system`, ADR-0013).** The neo-brutalist tokens/primitives and the `webinar-card` unit are already on `main` (#510 → #512–515; the card graduated in 004). No new DS primitive is expected; any bespoke element the `my-events.dc.html` canvas needs beyond the DS inventory runs the `build-ui-from-design-system` registry gate first and is recorded in the PR.
- **Dev-stand Postgres + Zitadel.** The registration e2e tests run against dev-stand Postgres (`DATABASE_URL`) and the guest-through-auth E2E against the live stand's `idp` per `.claude/rules/dev-stand.md`; endpoints/ports are read from `.env.local`.
