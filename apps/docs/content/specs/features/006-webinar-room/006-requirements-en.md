# 006 — Webinar room: embed player, live chat, heartbeat presence (Requirements)

> Epic: [Webinars — product brief](../../product/webinars/brief.md) · Wave 1, approved variant A «thin vertical». **The MVP-critical surface for the 2026-07-17 live webinar.** PRD source: [`006-product.md`](./006-product.md) (US-1…US-7). Mockup source of truth: the vendored canvas in [`design-source/`](../../../../../../design-source/README.md) — `webinar-room.dc.html` (desktop `1fr 400px` player + chat aside; mobile full-bleed player + tabs).

## Outcomes

- A **registered doctor** enters the room from the event page (004) or «мои события» (005) while the webinar is `live` and watches the stream **without any platform-caused obstacle** — the legacy "could not join the broadcast" class is retired (US-1).
- The player is instantiated from the event's **explicit provider enum** (`rutube | youtube | vk | cdnvideo`), configured in admin (007) — never inferred from the stream URL; we build **no** transcoding and **no** media server (US-1).
- When the embedded stream **fails to start playing** (a provider CDN stall, an embedding-disabled video), the doctor never faces a **silent black frame**: the room shows a **truthful in-frame status**, **auto-retries** the embed, and offers an in-room **«Перезапустить плеер»** affordance — never a full page reload and never a link off the platform (verified in-room presence is the sponsor value); presence keeps counting throughout (US-1).
- A doctor **reads and posts** in a live chat alongside the stream and sees others' messages appear **without reloading** — over Centrifugo, already in our stack (US-2).
- Presence is captured **from minute one** via a **server-authoritative heartbeat** — while the room tab is the visible, active tab the room posts an authenticated signal every N seconds and the backend appends it to a durable Postgres table (a backgrounded tab pauses the loop, so its minutes do not count) — with **no** doctor-facing "prove you're here" action (US-3).
- The captured data yields **actual per-doctor presence minutes** for the sponsor; concurrent tabs do not inflate a doctor's minutes; for the first webinar the sponsor report is a **manual export** from that data (US-3, US-4).
- Room access is **enforced server-side** — authenticated **and** registered only, with player config, chat access, and heartbeat acceptance all behind the gate. An unauthenticated visitor is guided to auth (003); an unregistered doctor to register (005); a non-`live` event shows the truthful lifecycle state (004) — never a soft UI wall over the player (US-5).
- The room is built from `@ds/design-system` tokens to the vendored canvas geometry (neo-brutalist language), verified at both breakpoints × both themes (ADR-0013).
- The doctor switches the room between **light and dark** with the room-header toggle — backed by a **portal-wide theme mechanism** (`.dark` on `<html>`, system `prefers-color-scheme` default, persisted user override always wins, no flash of the wrong theme); the visible toggle ships only in the room header, its placement on other portal surfaces being the tracked #510 unified-portal-chrome deferral (US-6).
- Before the **first** room entry the doctor is asked **once** for «Имя и фамилия»; the room header then renders an **initials avatar from the real display name** — stored on the `users` mirror, never faked from an email or a placeholder. The **same name authors the doctor's live-chat messages**, shown to every participant (with the non-PII participant tag as the fallback when unset); the JIT prompt discloses this up front. Registration stays untouched (US-7).

## Scope

**In:**

- **Server-side room admission gate** — the room content (player config, a chat connection credential, heartbeat acceptance) is served only to an **authenticated, registered** doctor while the event is `live`, via a server-issued `RoomAccess` grant; a direct URL, a shared link, or a crafted request that fails any condition is refused server-side (EARS-1).
- **Room composition + embed player** — the room renders to the `webinar-room.dc.html` composition (desktop `1fr 400px` player + chat aside; mobile full-bleed player + Чат / О эфире tabs) and instantiates the player from the event stream config's **explicit provider enum** `rutube | youtube | vk | cdnvideo`, never URL-sniffing; an unknown/absent provider yields a truthful "stream unavailable" room state (EARS-2).
- **In-room player failure states — watchdog + truthful status + in-room retry** — a provider-agnostic watchdog detects an embed that never starts playing (the cross-origin iframe is opaque, and `iframe.onload` fires even on a provider error page); on failure the room shows a truthful in-frame status, auto-retries a bounded number of times, then offers an in-room «Перезапустить плеер» affordance that re-creates the embed (never a full page reload, never an off-platform link), and clears the status on recovery — where a provider exposes a parent-observable embed API (youtube, rutube) the room layers its playing/error signals on top of the watchdog (EARS-18).
- **Live chat over Centrifugo** — a gated doctor reads the room chat and posts messages that fan out to every participant in real time without a reload; chat is available while the room is open (EARS-3).
- **Server-authoritative heartbeat presence capture** — while a gated doctor is in a live room **with the room tab visible and active**, the client posts an authenticated heartbeat every **N** seconds (server-side config, default **60 s**) — a **backgrounded tab (`document.hidden`) emits none**, and the loop resumes when the tab is visible again — and each accepted beat appends to a durable append-only Postgres table carrying `(doctor, event, instant)` (EARS-4).
- **Per-doctor presence-minute derivation** — the append-only beats yield actual per-doctor minutes, computed from the beat timestamps **parameterized over N**; concurrent tabs for the same doctor do not inflate the minutes; the data is sufficient for the first webinar's sponsor report by **manual export** (no report UI) (EARS-5).
- **Denied-access routing (the front door)** — an unadmissible caller is routed truthfully: unauthenticated → auth (003) → re-evaluated; authenticated-but-unregistered → guided to register (005) → admitted on success; event not `live` → the truthful 004 lifecycle state (EARS-6).
- **Room-close stops capture** — when the event leaves `live` (the director closes the room, 007), the server stops accepting heartbeats and chat posts for that event, and the room degrades to the truthful ended state (EARS-7).
- **Room read + write side** in `apps/api` — the `RoomConfig` read model, the `PostChatMessage` command (Centrifugo publish), and the `RecordPresenceHeartbeat` command with its durable presence table, all classified **`authenticated` `doctor_guest` `policy`** (registration-and-live gated) in the endpoint-authz matrix (ADR-0001 §2). Consumes the 005 `EventRoster`, the 004/007 `EventLifecycleState`, and the 007 stream config (EARS-8).
- **Embed-frame boundary** — the room embeds the external stream as a frame only; it does not transcode, re-host, proxy, DRM-sign, or record the stream, and emits no player-level telemetry (EARS-9).
- **RU-only copy from the typed message catalog** — no hardcoded user-facing strings, reusing the i18n-ready structure established in 003 (EARS-21) and reused in 004 / 005; any absolute date/time (the «О эфире» program schedule) presented in `Europe/Moscow` labeled **МСК** (EARS-10).
- **Canvas-faithful room** built from `@ds/design-system` tokens to `webinar-room.dc.html` at both breakpoints × both themes (EARS-11).
- **Portal theme mechanism + room-header toggle** — a portal-wide light/dark mechanism (`.dark` class on `<html>`, an inline FOUC-guard script, `localStorage` persistence, default = system `prefers-color-scheme`, an explicit user choice wins) whose **only** visible control is the webinar-room-header toggle — the canvas **44×44 icon-button**, never the DS form switch; the portal axe accessibility e2e extends to **both** themes (EARS-12, EARS-13).
- **Just-in-time display name + initials avatar + named chat authorship** — before the first room entry a doctor without a display name is prompted once («Имя и фамилия»; empty/whitespace-only rejected); the accepted name is written via an authed endpoint to a display-name column on the `users` mirror (the SSOT), and the room header renders an avatar with the initials of that real name. The **same name authors the doctor's live-chat messages** — every participant sees it as the message author (falling back to the non-PII «Участник <tag>» participant label when the poster has no name set); the JIT prompt discloses this up front («Ваше имя будут видеть участники чата эфира») (EARS-14, EARS-15, EARS-16, EARS-17).

**Explicitly out** (each a named deferral, not a silent default):

- **Director open/close of the room + stream-config authoring** — opening/closing the live room and configuring the stream (provider enum + embed reference) are owned by **feature 007** (admin/Refine). 006 **reads** the resulting `EventLifecycleState` and stream config to gate and to instantiate the player, and is built + E2E-driven against **seeded live events with seeded stream config + a seeded roster** until 007 lands (Dependencies & sequencing). This is the 006↔007 tracked seam, exactly as 005 carried its 007 dependency.
- **Question-to-lecturer («Вопросы» tab) + polls + presence-confirmation widgets («титровальные объекты»)** — the canvas shows a **Вопросы** tab and «Задать вопрос» CTAs; those are **wave 2** and are **not built** in 006 (the mobile tab set in wave 1 is Чат / О эфире; the desktop aside is chat only). This mirrors how 005 deferred the `my-events` Записи / Сертификаты tabs.
- **Auto-NMO accrual (90 min + 2 confirmations) and the auto sponsor report «Отчёт партнёра V2»** — wave 2. 006 only **captures** the presence data those verticals will consume; the wave-1 sponsor report is a manual export from it.
- **Certificate artifact** (NMO points vs downloadable PDF) — wave 2 owner call; out of scope here (PRD open question, left for wave 2).
- **Sponsor report V2 exact shape** (columns/joins) — wave 2; the wave-1 manual export needs only the raw per-doctor minutes (PRD open question, wave 2).
- **SDN Player** — **not** in the provider enum. The closed list is `rutube | youtube | vk | cdnvideo` — the RU-reachable, embeddable providers (Rutube the standing default for RU-audience events; #1154). Extending the list to a further provider later is an additive **migration** (`ALTER TYPE … ADD VALUE` + a new embed branch + its `EMBED_REF_SHAPES` entry), not a shape 006 pre-builds (owner decision 2026-07-06).
- **Video transcoding, own media server, DRM / signed playback, player-level telemetry** — out of the epic's embed frame (EARS-9).
- **Recordings archive** — wave 2.
- **The public event page + listing (004), the registration write + «мои события» (005), and the auth flow (003)** — consumed, not owned here.
- **Theme toggle on other portal surfaces** — the theme mechanism is portal-wide, but the visible control ships only in the room header (per the canvas); placing the control across the rest of the portal chrome is **deferred to the #510 unified-portal-chrome re-skin** — a named, tracked deferral, not a silent seam.
- **Name collection at registration** — registration keeps collecting exactly its current fields (owner decision 2026-07-11 — zero added funnel friction on live prod); the display name is collected just-in-time at first room entry, never at registration. **In scope, not out:** the display name IS shown to other chat participants as the author of a doctor's live-chat messages (owner decision 2026-07-23, Option A) — a reversal of the earlier «visible only to you» stance; the JIT prompt discloses this before the name is entered. Retroactive renaming of pre-feature chat history is technically impossible (the old tag is a one-way hash) and is not attempted.

## Constraints

- **The gate is server-side, not a UI wall.** Admission (authenticated ∧ registered ∧ `live`) is enforced by the backend before any room content — player config, chat credential, or heartbeat acceptance — is served. A direct room URL, a shared link, or a crafted request never renders the player or is admitted; the legacy soft-wall-over-the-player pattern is banned (PRD acceptance; recon §6).
- **Registration is the admission basis, reused not reimplemented.** The registered check reads the 005 `EventRoster` for `(doctor, event)`; 006 adds **no** registration primitive and creates no registration. An unregistered doctor is routed to 005, not admitted (EARS-1, EARS-6).
- **Player selection is an explicit enum, never URL-sniffing.** The player provider is a closed list `rutube | youtube | vk | cdnvideo` stored in the event stream config (authored in 007); the room instantiates the embed from the enum value, never by inspecting the URL string. An unknown/absent provider is a truthful "stream unavailable" state, not a guessed embed (owner decision 2026-07-06; the legacy URL-sniffing mistake, recon §5).
- **A stream failure is truthful and in-room — never a black frame, a page reload, or an off-platform escape.** A cross-origin embed can fail to start playing (a provider CDN stall, an embedding-disabled video) while `iframe.onload` still fires, so the room runs a provider-agnostic watchdog (a named design constant, ~20 s) as the detection floor and layers provider-observable signals on top where they exist (youtube, rutube). On failure the remedy stays **in the room**: a truthful status, bounded auto-retry, then a manual «Перезапустить плеер» that re-creates the embed — **never** a full page reload, and **never** a "watch on the provider's site" link. Verified in-room presence is the sponsor value; an off-platform link leaks that audience away (owner decision 2026-07-22). Player state does not touch presence capture (EARS-18).
- **Heartbeat cadence N is server-side config, default 60 s.** The presence-minute math is **parameterized over N** — an operator-confirmed different cadence changes **config**, not the spec or the code. The client's post interval is driven by the same server-provided value (owner decision 2026-07-06; legacy 60 s evidence, recon §10-3).
- **Presence tracks the visible, active tab only.** The client heartbeat loop is gated on the Page Visibility API: while the room tab is backgrounded (`document.hidden`), the client emits **no** beats, so a backgrounded tab's minutes do not count toward the sponsor report; the loop resumes when the tab becomes visible again. This is a **client-side capture gate** — the server still refuses any beat from an ungated caller or a closed room. Tab-coalescing (EARS-5) is unchanged: two _visible_ sessions in the same interval still count once. NMO thresholds and interactive presence confirmations remain wave 2 (owner decision 2026-07-07).
- **Presence is server-authoritative and durable.** Beats are accepted only from an authenticated, gated doctor and appended to a durable Postgres table (never a client-trusted count, never the exposed-service-key client pings the legacy used, recon §6). Centrifugo presence is ephemeral and is **not** the durable record — the minutes come from the append table (brief adopt-vs-build).
- **Concurrent tabs must not inflate minutes.** Beats from a doctor's parallel sessions for the same event coalesce into one presence timeline; two tabs open for the same minute count as one minute of presence (PRD acceptance).
- **Heartbeats and chat posts stop when the room closes.** Once the event leaves `live` (room closed by 007), the server refuses further beats and posts for that event; minutes are computed over the window the room was open (EARS-7).
- **Embed frame only.** No transcoding, media server, DRM/signed playback, re-hosting, or player telemetry — the room composes a configured iframe/player around the sponsor's external stream, nothing more (epic scope; the Mediator.cloud buy-option was rejected).
- **Chat rides Centrifugo, already in the stack.** No new realtime infrastructure decision — Centrifugo is provisioned on the dev stand and in the engineering-readiness spec; 006 adds a room channel + a scoped connection token, not a new transport (brief adopt-vs-build; dev-stand.md).
- **No hardcoded origin / URL / stream reference.** The portal origin, API endpoints, Centrifugo URL, and the stream reference are read from configuration / the event record, never hardcoded in code or spec (AGENTS.md §9; dev-stand.md).
- **Canonical timezone.** Any absolute date/time rendered in the room (the «О эфире» program schedule) is presented in `Europe/Moscow` labeled **МСК**, computed from the read model's canonical instant, and must not drift to the viewer's local timezone (mirrors 004 EARS-12 / 005 EARS-11). The live-elapsed indicator («В эфире · N мин») is derived from the event's **actual go-live instant** (`live_at`, stamped by 007 `OpenRoom` on the `published → live` transition) — **never** the scheduled `startsAt` (a broadcast that starts late must not show inflated elapsed minutes); a legacy `live` row with no `live_at` renders the pill with no minute suffix (truthful, never back-filled from the schedule).
- **Canvas is the fidelity spec (ADR-0013 / ADR-0014 §4).** The room is built from `@ds/design-system` tokens to the vendored `webinar-room.dc.html` geometry (2px borders, `6px 6px 0` shadow on the player + chat aside, the live pulse badge, the desktop `1fr 400px` grid, the mobile tabbed layout). Arbitrary Tailwind values are lint-blocked; where prose and the canvas disagree, the canvas wins.
- **Stack** (ADR-0002 / ADR-0004): the room-config read + the chat and heartbeat commands are NestJS + `nestjs-zod`, schema SSOT in `packages/schemas/`, URI versioning `/v1/...`, Vitest + supertest; the portal is Next.js 15. Service-dependent tests `skipIf` their dependency env (`DATABASE_URL`, `IDP_ISSUER`, `CENTRIFUGO_URL`) is absent so they do not redden the shared CI unit job.
- **The theme mechanism is portal-wide; the control is room-only (for now).** `.dark` is applied on `<html>` **before first paint** by an inline FOUC-guard script; an explicit choice persists in `localStorage` (`ds-theme`) and always wins over the system value; absent a stored choice the portal follows the system `prefers-color-scheme` live. The `@ds/design-system` dark tokens already ship (`semantic.dark.json` + `.dark`-scoped CSS) — 006 wires the class application they have been waiting for. The control is the canvas **44×44 icon-button** in the room header (glyph ☾ in light / ☀ in dark; design §10), bordered with the on-header `header-hairline` token — the DS `switch.tsx` stays the form switch primitive, not the theme control (owner decision 2026-07-12). The webinar-room header is the only surface with a visible toggle; extending the control across portal chrome is the tracked #510 deferral (owner decision 2026-07-11).
- **Display name: JIT-only collection, self-only exposure.** The display name is collected exactly once, at room entry — never at registration (owner decision 2026-07-11 — zero added funnel friction on live prod). Its SSOT is a display-name column on the `users` mirror (003), written via an authed self-scoped endpoint; the Zitadel profile placeholder stays never-read. It is served **only** to the doctor themself (their own session); the room-header avatar derives its initials from this real name — email-derived or placeholder-derived fakes are forbidden — and chat identity stays the non-PII author tag, unchanged.

## Event Model

Feature 006 is the **room vertical**: a `doctor_guest`-authenticated, registration-and-live-gated read (`RoomConfig`) plus two gated commands (`PostChatMessage`, `RecordPresenceHeartbeat`) with a durable append-only presence table. It **reads** the 005 `EventRoster` (admission basis) and the 004/007 `EventLifecycleState` + stream config; it **produces** the presence beats the sponsor report draws from. It owns **no** auth primitive (reuses 003), **no** registration (reads 005), and **no** event authoring / room open-close (reads 007). It additionally owns one self-scoped write — `SetDisplayName` onto the `users` mirror (the JIT room-entry prompt, EARS-14). The theme mechanism is client-side only (`<html>` class + `localStorage`) and has **no** event-model footprint.

### Commands (handled by `apps/api`, `doctor_guest`-authenticated + registration-and-live `policy` gate)

- `PostChatMessage(idOrSlug, text)` — post a chat message to the room channel; the backend authorizes the gate, then publishes to Centrifugo. Refused if the room is not open (event not `live`) or the caller is not gated.
- `RecordPresenceHeartbeat(idOrSlug)` — append one presence beat for `(doctor, event, instant)`. Refused if the room is not open or the caller is not gated. Idempotent within an interval (concurrent tabs coalesce; EARS-5).
- `SetDisplayName(displayName)` — write the caller's display name to the `users`-mirror column (`authenticated` `doctor_guest` `fast-path` — self-scoped, no room gate). Rejects an empty or whitespace-only value; invoked once from the JIT room-entry prompt (EARS-14), after which the prompt never reappears.

### Events

| Event                                                 | Owner                       | Notes                                                                                                                                                             |
| ----------------------------------------------------- | --------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `PresenceHeartbeatRecorded`                           | `apps/api`                  | One append-only row per accepted beat `(doctor, event, instant)`. The durable basis for per-doctor presence minutes (US-3, US-4). Refused after room close.       |
| `ChatMessagePosted`                                   | `apps/api` (via Centrifugo) | A gated doctor's message fanned out over the room channel in real time (US-2). Not the durable presence record.                                                   |
| `EventWentLive` / `EventEnded` (room opened / closed) | 007 (consumed read-only)    | 006 reads the resulting `EventLifecycleState`: `live` ⇒ the room is open (beats + posts accepted); leaving `live` ⇒ the room is closed (beats + posts refused).   |
| `DoctorRegisteredForEvent`                            | 005 (consumed read-only)    | 006 reads the resulting `EventRoster` to decide admission for `(doctor, event)`. 006 creates no registration.                                                     |
| `DisplayNameSet`                                      | `apps/api`                  | The doctor's display name written to the `users` mirror (US-7). Self-scoped; never fanned out to other participants — chat identity stays the non-PII author tag. |

### Read models

- **`RoomConfig`** — per gated caller, per live event: `{ provider ∈ {rutube, youtube, vk, cdnvideo}, embedRef, chatToken (Centrifugo, room channel), heartbeatIntervalSeconds (= N) }`. Served **only** to an authenticated, registered doctor for a `live` event; never to a guest, an unregistered doctor, or for a non-`live` event.
- **`EventPresence`** — the append-only beats for an event; the derived **per-doctor presence minutes** (parameterized over N, concurrent-tab-coalesced) are the sponsor deliverable (manual export in wave 1). Never exposed on a public surface; no registrant PII on a public endpoint (recon §6).
- **`EventRoster`** (owned by 005, **consumed** here) — the set of current registrations, the admission basis; **`EventLifecycleState`** + **stream config** (owned by 007, surfaced by 004, **consumed** here) — the live check + the provider enum/embed reference.
- **`MyDisplayName`** — the caller's **own** display name (unset until the JIT prompt completes), served only to the authenticated doctor's own session; the portal reads it to decide the one-time room-entry prompt and to derive the header-avatar initials. Never served for another user (EARS-16).

### Policies

- **On `RoomConfig` read / `PostChatMessage` / `RecordPresenceHeartbeat`** → admit iff `authenticated ∧ registered (005 roster) ∧ event live`; else refuse server-side and route (EARS-6). No soft UI wall serves content to an ungated caller.
- **On `RecordPresenceHeartbeat`** → append one beat; concurrent tabs for the same `(doctor, event)` coalesce into one presence timeline (no minute inflation; EARS-5). Beats are refused once the room is closed (EARS-7).
- **On the event leaving `live`** (room closed, 007) → refuse further beats and posts for that event; minutes are computed over the open window (EARS-7).
- **On room entry (portal)** → if the gated doctor's `MyDisplayName` is unset, present the one-time «Имя и фамилия» prompt before rendering the room; on `SetDisplayName` success render the room (EARS-14). Once a name is set the prompt never reappears. This is a portal pre-entry step — the server admission gate stays exactly `authenticated ∧ registered ∧ live`.

## EARS requirements

> **Numbering convention:** flat (`EARS-1`, `EARS-2`, …) per ADR-0006 §4. EARS-1…7, EARS-12, EARS-14…15, EARS-17, and EARS-18 are the functional handlers (each becomes a child Issue); EARS-8…11, EARS-13, and EARS-16 are cross-cutting ubiquitous / unwanted-behavior requirements enforced across the surface. A handler with several shall-clauses is numbered `N.M` (EARS-18 carries 18.1…18.5). Each clause carries a `realizes: US-N` backlink to the PRD story it formalizes (ADR-0014 §2).

**Room admission & composition**

- **EARS-1** _(realizes: US-1, US-5)_ — When a caller requests room content for an event (the player config, a chat connection credential, or heartbeat acceptance), the system shall serve it **only if** the caller is **authenticated AND registered for that event (005 roster) AND the event is `live`**, issuing a server-side `RoomAccess` grant; a direct room URL, a shared link, or a crafted request that fails any of the three conditions shall be **refused server-side** — there is no soft UI wall that renders the player, chat, or accepts a beat for an ungated caller.
- **EARS-2** _(realizes: US-1)_ — Where a `RoomAccess` grant exists, the system shall render the room to the `webinar-room.dc.html` composition (desktop `1fr 400px` player + chat aside; mobile full-bleed player + Чат / О эфире tabs) and **instantiate the embed player from the event stream config's explicit provider enum** — a closed list `rutube | youtube | vk | cdnvideo` — **never** by inspecting or sniffing the stream URL; an unknown or absent provider shall yield a truthful "stream unavailable" room state, not a guessed embed.

**Live chat**

- **EARS-3** _(realizes: US-2)_ — Where the room is open, the system shall let a gated doctor **read the live chat and post messages over Centrifugo** (the room channel keyed by event id), fanning each posted message out to every participant **in real time without a reload**; the session's chat is available while the room is open, and a message is posted only through the gated command (no client can publish to the room channel without the server-issued, gate-scoped token).

**Presence capture**

- **EARS-4** _(realizes: US-3)_ — While a gated doctor is present in a live room **and the room tab is the visible, active tab** (Page Visibility API — `document.hidden` is false), the client shall post an **authenticated heartbeat every N seconds** — N is a **server-side config, default 60 s**, delivered to the client in `RoomConfig` — and the system shall append each accepted beat to a **durable append-only Postgres table** carrying `(doctor, event, instant)`; **while the room tab is backgrounded (`document.hidden` is true) the client shall emit no heartbeats — a backgrounded tab's minutes do not count toward the sponsor report — and the heartbeat loop shall resume when the tab becomes visible again**. Presence is captured from minute one with **no** doctor-facing "prove you're here" action.
- **EARS-5** _(realizes: US-3, US-4)_ — The append-only beats shall yield **actual per-doctor presence minutes** for an event, computed from the beat timestamps **parameterized over N** (an operator-confirmed different cadence changes config, not the spec or code); **concurrent tabs for the same doctor shall not inflate** that doctor's minutes (beats from parallel sessions coalesce into one presence timeline); and the captured data shall be **sufficient to produce the first webinar's sponsor report by manual export** — no report UI in wave 1. The **live in-room presence count** (the distinct doctors currently present in the open window) shall be **server-authoritative and pushed in realtime**: when an accepted beat or a presence-window expiry **changes** that distinct-doctor count, the system shall **publish the recomputed count to the room's realtime channel**, and clients shall **render the published value instantly** — without waiting on the observer's own next beat; while the realtime channel is **unavailable** the counter shall **degrade to the heartbeat-ack refresh path** (the count returned on each accepted beat) and **never freeze silently on a stale value**. The count source is this server-side aggregate, **never Centrifugo native channel presence** (which counts connections/tabs, not distinct doctors, and feeds the sponsor attendance report); the 60 s heartbeat cadence and the sponsor minutes/report derivation are unchanged.

**Access branches & room close**

- **EARS-6** _(realizes: US-1, US-5)_ — When a caller reaches the room but is **not admissible**, the system shall route them **truthfully** rather than show a soft wall: an **unauthenticated** visitor is taken through the auth flow (003) and re-evaluated; an **authenticated but unregistered** doctor is guided to register (005) and admitted on success; an event that is **not `live`** (not yet opened or already closed) shows the truthful lifecycle state (004 page states) with no watchable room.
- **EARS-7** _(realizes: US-3, US-4)_ — When the event leaves `live` (the director closes the room, feature 007), the system shall **stop accepting heartbeats and chat posts** for that event — a late beat or post is refused server-side — and the room shall degrade to the truthful ended state; per-doctor presence minutes are computed over the beats captured **while the room was open**.

**Cross-cutting (ubiquitous / unwanted-behavior)**

- **EARS-8** _(realizes: US-5)_ — The `RoomConfig` read, the chat connection/publish endpoint, and the heartbeat endpoint shall be classified **`access: authenticated`, `required_roles: doctor_guest`, `auth_check: policy`** in the endpoint-authz matrix (ADR-0001 §2) — the registration-and-live gate is a policy evaluation beyond a role fast-path. **No** endpoint serves room content, a chat credential, or accepts a beat for an unauthenticated or unregistered caller; **no** doctor ever receives another doctor's presence data or the roster (a chat message legitimately carries its own poster's display name to participants per EARS-17 — that is the poster's chosen chat authorship, not a leak of another doctor's private data); and the per-doctor presence data is never exposed on a public surface.
- **EARS-9** _(realizes: US-1)_ — The system shall embed the external stream as a **frame only**: it shall **not** transcode, re-host, proxy, DRM-sign, or record the stream, and shall **not** emit player-level telemetry — the room composes a configured iframe/player around the sponsor's stream, nothing more (the epic embed-frame boundary; the Mediator.cloud buy-option was rejected).
- **EARS-10** _(realizes: US-1, US-2, US-3)_ — The 006 room surfaces shall render in **Russian** with **no hardcoded user-facing strings** — all copy (the live badge, room chrome, chat placeholder/labels/empty-state, the access-branch guidance, and the "stream unavailable"/ended states) sourced from the typed message catalog established in 003 (EARS-21) and reused in 004 / 005; any **absolute** date/time (the «О эфире» program schedule) is presented in `Europe/Moscow` labeled **МСК**, and the live-elapsed indicator is derived from the event's **actual go-live instant** (`live_at`), not the scheduled `startsAt`.
- **EARS-11** _(realizes: US-1, US-2)_ — The room shall be built from **`@ds/design-system` tokens** to the vendored `webinar-room.dc.html` geometry (2px borders, `6px 6px 0` shadow on the player + chat aside, the live pulse badge, the desktop `1fr 400px` grid, the mobile tabbed layout), rendering correctly at **both breakpoints × both themes**; arbitrary Tailwind values are lint-blocked (ADR-0013), and any bespoke element beyond the DS inventory (the embed player frame, the chat panel) runs the `build-ui-from-design-system` registry gate first (shadcn chat primitives / react-player + thin Rutube iframe per the epic adopt-vs-build) and is recorded in the PR.

**Room theme (header toggle)**

- **EARS-12** _(realizes: US-6)_ — When a gated doctor activates the room-header theme toggle, the system shall switch the portal between light and dark by toggling the **`.dark` class on the `<html>` element** and shall **persist the choice in `localStorage`** (key `ds-theme`); where no persisted choice exists, the portal shall follow the system **`prefers-color-scheme`**, and a persisted explicit choice shall win over the system value; on every page load an **inline FOUC-guard script** shall apply the resolved theme **before first paint** — the page never flashes the wrong theme. The toggle shall render as the canvas **44×44 icon-button** in the room-header icon-button family (transparent background, 2px `header-hairline` border, full-strength header-foreground glyph — **☾ in light / ☀ in dark** — hover raising the border to full strength), a `<button>` carrying `aria-pressed` for the dark state, the accessible name «Переключить тему» from the message catalog, and the DS focus ring on focus-visible — never the DS form switch. The visible toggle ships **only** in the webinar-room header (per the canvas); its placement on other portal surfaces is the tracked **#510** unified-portal-chrome deferral.
- **EARS-13** _(realizes: US-6)_ — The portal accessibility gate shall exercise **both themes**: the portal axe e2e (`playwright-axe`) drives its covered surfaces in light **and** dark, and a dark-rendered surface shall introduce no new axe violations relative to light.

**Display name & avatar**

- **EARS-14** _(realizes: US-7)_ — When a gated doctor whose display name is unset enters the webinar room for the first time, the portal shall prompt **once** for «Имя и фамилия» before rendering the room; the prompt copy shall **disclose that the name is shown to other chat participants** («Ваше имя будут видеть участники чата эфира»), replacing the earlier «видно только вам» promise; the system shall **reject** an empty or whitespace-only value with a truthful error, and on acceptance shall write the trimmed name via the authed `SetDisplayName` endpoint to the display-name column on the `users` mirror (the SSOT); once a display name is set the prompt shall never reappear, and registration shall keep collecting exactly its current fields — nothing new.
- **EARS-15** _(realizes: US-7)_ — Where a gated doctor's room renders, the room header shall display an avatar carrying the **initials derived from that doctor's real display name** (the first letters of the first and last words of the saved name, uppercased; a single-word name yields a single initial); the header shall **never** fabricate initials from an email address, the never-read Zitadel placeholder profile, or any other stand-in — where no display name exists, no fabricated avatar is rendered.
- **EARS-16** _(realizes: US-7)_ — The self-profile **read/write of the display name** shall be served **only to the doctor themself** (their own authenticated session): the `SetDisplayName` write and the `GET /v1/me/display-name` self-read are classified `access: authenticated`, `required_roles: doctor_guest`, `auth_check: fast-path` in the endpoint-authz matrix (ADR-0001 §2); **no** profile-read endpoint exposes another user's display name, and the Zitadel profile placeholder stays **never-read**. The name's participant-visible surface is exactly one: the doctor's **own** live-chat authorship (EARS-17) — the poster's name authors the poster's own messages, never fetched for or attributed to anyone else.
- **EARS-17** _(realizes: US-7)_ — Where a gated doctor posts a chat message in a live room, the system shall stamp the fanned-out payload with the poster's **own display name** in an `authorName` field (the `users`-mirror `display_name`), shown to every participant as the message author (owner decision 2026-07-23, Option A — a reversal of the earlier «visible only to you» stance, shown immediately with no re-consent round). Where the poster has no display name set — and for **legacy history** minted before this field existed — `authorName` is `null` and the portal falls back to the non-PII «Участник <tag>» participant label; the field is nullable so no migration or backfill is needed. The name carried is **only** the poster's own real, saved name — never fabricated from an email address, roster identity, or the never-read Zitadel placeholder (EARS-15 spirit); the stable non-PII `authorTag` still rides every payload as the self-identity key the poster's client matches its own `selfTag` against.

**In-room player failure states (watchdog + truthful status + in-room retry)**

- **EARS-18** _(realizes: US-1)_ — the in-room player-failure handler: a stream that mounts but never starts playing must degrade to a truthful, retryable in-room state, never a silent black frame and never an escape off the platform. It carries five shall-clauses:
  - **EARS-18.1 — Watchdog (detection floor).** While a room has a stream config and the embed player has mounted, when **no playing signal is observed within the watchdog threshold** (a named design constant, default ~20 s, `§3.1`) after mount, the system shall present a truthful in-frame **"stream not loading" status overlay** in the player region (canvas-styled, copy from the typed catalog per EARS-10) — never a silent black frame. The watchdog is provider-agnostic because a cross-origin iframe is opaque and `iframe.onload` fires even on a provider error page.
  - **EARS-18.2 — Provider-event layering.** Where the provider embed exposes a **parent-observable API** (youtube via the IFrame Player API with `enablejsapi=1`+`origin`; rutube via its postMessage JSON API), the system shall consume it to **clear the watchdog on the observed playing signal** and to surface provider error states; for YouTube error codes it shall distinguish **"embedding disabled by the broadcaster" (101/150)** from **"video unavailable" (100)** in the status text. Where the provider exposes **no** parent-observable API for the live embed — **vk** (its `js_api` does not work for live broadcasts) and **cdnvideo** (Clappr, same-origin script only, no postMessage) — failure detection is **watchdog-only**; this is a stated capability constraint of those providers, not a deferral.
  - **EARS-18.3 — Bounded retry, in-room only.** On a failure state the system shall **auto-retry the embed a bounded number of times** (default up to 2 attempts, with a delay between attempts) and, if still failed, shall offer a manual **«Перезапустить плеер»** affordance that **re-creates the embed** (or performs an in-place provider-API reload where supported — rutube `changeVideo`, YouTube `loadVideoById`) — **never** a full page reload, and **never** a "watch on the provider's site" / off-platform link (owner decision 2026-07-22).
  - **EARS-18.4 — Recovery clears the overlay.** When a **playing signal is observed after a failure state** — from an auto-retry, the manual restart, or the provider self-recovering (e.g. a CDN failover) — the system shall clear the status overlay and present the playing stream.
  - **EARS-18.5 — Presence is unaffected by player state.** The player failure/retry/unavailable state shall **not** affect presence capture: the visibility-gated heartbeat loop (EARS-4) and the in-room presence count (EARS-5) are decoupled from player state — a failed, retrying, or unavailable player neither pauses nor resets the heartbeat, and the doctor remains counted present while the room tab is visible.

## Invariants

- Room content (player config, chat credential, heartbeat acceptance) is served **only** to an authenticated, registered doctor for a `live` event — no path (direct URL, shared link, crafted request) admits an ungated caller; there is no soft UI wall (EARS-1, EARS-8).
- The player provider is always one of the **closed enum** `rutube | youtube | vk | cdnvideo`, read from the event stream config — never inferred from the URL; an unknown/absent provider is a truthful "stream unavailable" state, never a guessed embed (EARS-2).
- A stream that fails to start playing is **never a silent black frame**: a provider-agnostic watchdog (with provider-observable signals layered on top for youtube/rutube) surfaces a truthful in-frame status, the room auto-retries and offers an in-room «Перезапустить плеер», and the remedy is never a full page reload or an off-platform link; player state never affects presence capture (EARS-18).
- The client heartbeat loop is **visibility-gated** — a backgrounded room tab (`document.hidden`) emits **no** beats, so only the visible, active tab's presence is captured; the loop resumes on re-visibility (EARS-4).
- Every presence beat is **server-authoritative and durable** — appended to Postgres for an authenticated gated doctor, never a client-trusted count and never via an exposed key (EARS-4).
- Concurrent tabs for the same `(doctor, event)` **never inflate** presence minutes — parallel beats coalesce into one presence timeline (EARS-5).
- Beats and chat posts are accepted **only while the room is open** (`live`); once the room closes they are refused, and minutes are computed over the open window (EARS-7).
- Presence minutes are **parameterized over N** (server config, default 60 s) — a different confirmed cadence changes config, not the spec or code (EARS-4, EARS-5).
- The per-doctor presence data and the roster are **never** on a public surface — no registrant PII leaks to a public endpoint (EARS-8; recon §6).
- The room **embeds only** — it never transcodes, re-hosts, DRM-signs, records, or telemeters the stream (EARS-9).
- Any absolute date/time in the room is in `Europe/Moscow` labeled МСК, derived from one canonical instant (EARS-10).
- The theme is resolved **before first paint** (inline FOUC-guard, `.dark` on `<html>`); a persisted user choice always wins over the system `prefers-color-scheme`; the only visible toggle is the room header's canvas icon-button until #510 (EARS-12).
- The display name's SSOT is the `users`-mirror column; its self-profile read/write is served only to its owner — never fetched for another participant or exposed on a public surface. Its one participant-visible surface is the poster's **own** live-chat authorship (`authorName`, EARS-17), falling back to the non-PII `authorTag` when unset; the Zitadel placeholder profile stays never-read (EARS-14, EARS-16, EARS-17).
- Avatar initials always derive from the doctor's real, saved display name — never from an email address or a placeholder profile (EARS-15).

## Dependencies & sequencing

- **MVP-critical deadline — 2026-07-17.** 006 is the surface real doctors watch the live webinar on; it must run end-to-end on the new platform by the air date, with **zero platform-caused "could not join" complaints** (PRD acceptance; brief success metric). Sequencing below keeps 006 completable as its own vertical on seeds + real 003/004/005 so the deadline does not wait on 007.
- **Auth foundation (feature 003) — shipped.** The room's gate reuses 003's BFF session as the authenticated principal; an unauthenticated visitor is routed through the shipped 003 flow (EARS-6). 006 adds no auth primitive.
- **Public event page + join path (feature 004).** The room is entered from the 004 event page / join path; a non-`live` event falls back to the 004 lifecycle state (EARS-6). 006 is sequenced **after** 004 is on `main` (the join path exists to build on); until then the room is E2E-driven from a seeded live event page.
- **Registration record + `EventRoster` (feature 005).** Admission (EARS-1) reads the 005 `EventRoster` for `(doctor, event)`; an unregistered doctor is routed to 005 (EARS-6). 006 is sequenced **after** 005 is on `main` (the roster is the admission basis); this is the 005↔006 blocking link `open-ears-issues` wires (its step 4). Until then, admission is E2E-driven against a **seeded roster**.
