# 006 — Вебинарная комната: embed-плеер, живой чат, heartbeat-присутствие (Требования)

> Эпик: [«Вебинары» — продуктовый бриф](../../product/webinars/brief-ru.md) · Волна 1, утверждённый вариант A «тонкая вертикаль». **MVP-критичная поверхность для живого вебинара 2026-07-17.** Источник PRD: [`006-product-ru.md`](./006-product-ru.md) (US-1…US-5). Источник точности мокапа: завендоренный канвас в [`design-source/`](../../../../../../design-source/README.md) — `webinar-room.dc.html` (десктоп `1fr 400px` плеер + чат-панель; мобайл — full-bleed плеер + вкладки).

## Итоги (Outcomes)

- **Зарегистрированный врач** входит в комнату со страницы мероприятия (004) или из «мои события» (005), пока вебинар `live`, и смотрит стрим **без единого препятствия по вине платформы** — легаси-класс «не смог подключиться к эфиру» списан (US-1).
- Плеер инстанцируется из **явного enum провайдера** мероприятия (`rutube | youtube`), настраиваемого в админке (007) — никогда не выводится из URL стрима; мы строим **ни** транскодинга, **ни** медиасервера (US-1).
- Врач **читает и пишет** в живом чате рядом со стримом и видит чужие сообщения **без перезагрузки** — через Centrifugo, уже в нашем стеке (US-2).
- Присутствие фиксируется **с первой минуты** через **server-authoritative heartbeat** — пока вкладка комнаты видимая и активная, комната шлёт аутентифицированный сигнал каждые N секунд, бэкенд дописывает его в долговечную таблицу Postgres (фоновая вкладка ставит цикл на паузу, поэтому её минуты не считаются) — **без** действия врача «докажи, что ты здесь» (US-3).
- Зафиксированные данные дают **фактические минуты присутствия по каждому врачу** для спонсора; параллельные вкладки не завышают минуты врача; по первому вебинару отчёт спонсору — **ручная выгрузка** из этих данных (US-3, US-4).
- Доступ в комнату **принуждается на сервере** — только аутентифицированный **и** зарегистрированный, с конфигом плеера, доступом к чату и приёмом heartbeat'ов за воротами. Неаутентифицированного посетителя ведут в auth (003); незарегистрированного врача — зарегистрироваться (005); не-`live` мероприятие показывает правдивое состояние жизненного цикла (004) — никогда мягкой UI-стены поверх плеера (US-5).
- Комната собрана из токенов `@ds/design-system` по геометрии завендоренного канваса (нео-брутализм), проверена на обоих брейкпоинтах × обеих темах (ADR-0013).

## Scope

**В scope:**

- **Серверный гейт допуска в комнату** — содержимое комнаты (конфиг плеера, credential-подключение к чату, приём heartbeat'ов) отдаётся только **аутентифицированному, зарегистрированному** врачу, пока мероприятие `live`, через серверно выданный грант `RoomAccess`; прямой URL, пересланная ссылка или сконструированный запрос, не выполняющие любое из условий, отклоняются на сервере (EARS-1).
- **Композиция комнаты + embed-плеер** — комната рендерится по композиции `webinar-room.dc.html` (десктоп `1fr 400px` плеер + чат-панель; мобайл — full-bleed плеер + вкладки Чат / О эфире) и инстанцирует плеер из **явного enum провайдера** стрим-конфига `rutube | youtube`, никогда не URL-sniffing; неизвестный/отсутствующий провайдер даёт правдивое состояние «стрим недоступен» (EARS-2).
- **Живой чат через Centrifugo** — гейтированный врач читает чат комнаты и пишет сообщения, которые в реальном времени разлетаются всем участникам без перезагрузки; чат доступен, пока комната открыта (EARS-3).
- **Server-authoritative heartbeat-фиксация присутствия** — пока гейтированный врач в live-комнате **с видимой, активной вкладкой комнаты**, клиент шлёт аутентифицированный heartbeat каждые **N** секунд (server-side config, дефолт **60 с**) — **фоновая вкладка (`document.hidden`) не шлёт ничего**, а цикл возобновляется, когда вкладка снова видима, — и каждый принятый сигнал дописывается в долговечную append-only-таблицу Postgres с `(врач, мероприятие, момент)` (EARS-4).
- **Вывод минут присутствия по каждому врачу** — append-only-сигналы дают фактические минуты по каждому врачу, вычисленные из меток времени сигналов **параметризованно по N**; параллельные вкладки одного врача не завышают минуты; данных достаточно для спонсорского отчёта первого вебинара через **ручную выгрузку** (без UI отчётов) (EARS-5).
- **Роутинг отказа в доступе (парадная дверь)** — недопустимого вызывающего ведут правдиво: неаутентифицированный → auth (003) → переоценка; аутентифицированный-но-незарегистрированный → ведут зарегистрироваться (005) → впуск по успеху; мероприятие не `live` → правдивое состояние жизненного цикла 004 (EARS-6).
- **Закрытие комнаты останавливает фиксацию** — когда мероприятие выходит из `live` (режиссёр закрывает комнату, 007), сервер перестаёт принимать heartbeat'ы и сообщения чата по мероприятию, и комната деградирует в правдивое ended-состояние (EARS-7).
- **Read + write-сторона комнаты** в `apps/api` — read-модель `RoomConfig`, команда `PostChatMessage` (публикация в Centrifugo) и команда `RecordPresenceHeartbeat` с её долговечной таблицей присутствия, все классифицированы **`authenticated` `doctor_guest` `policy`** (гейт «зарегистрирован и live») в матрице endpoint-authz (ADR-0001 §2). Потребляет `EventRoster` из 005, `EventLifecycleState` из 004/007 и стрим-конфиг из 007 (EARS-8).
- **Граница embed-рамки** — комната встраивает внешний стрим как рамку и только: не транскодирует, не рехостит, не проксирует, не DRM-подписывает, не записывает стрим и не эмитит телеметрию уровня плеера (EARS-9).
- **RU-only копирайт из типизированного каталога сообщений** — без хардкод-строк, переиспользуя i18n-готовую структуру из 003 (EARS-21) и 004 / 005; любая абсолютная дата/время (расписание «О эфире») в `Europe/Moscow` с пометкой **МСК** (EARS-10).
- **Канвас-точная комната**, собранная из токенов `@ds/design-system` по `webinar-room.dc.html` на обоих брейкпоинтах × обеих темах (EARS-11).

**Явно вне scope** (каждое — именованная отсрочка, не молчаливый дефолт):

- **Режиссёрское открытие/закрытие комнаты + авторство стрим-конфига** — открытие/закрытие live-комнаты и настройка стрима (enum провайдера + ссылка на embed) принадлежат **фиче 007** (admin/Refine). 006 **читает** итоговый `EventLifecycleState` и стрим-конфиг, чтобы гейтировать и инстанцировать плеер, и строится + E2E-прогоняется на **seed-мероприятиях live с seed-стрим-конфигом + seed-списком**, пока 007 не приземлится (Зависимости и секвенирование). Это отслеживаемый шов 006↔007, ровно как 005 нёс свою зависимость от 007.
- **Вопрос лектору (вкладка «Вопросы») + опросы + виджеты подтверждения присутствия («титровальные объекты»)** — канвас показывает вкладку **Вопросы** и CTA «Задать вопрос»; это **волна 2** и в 006 **не строится** (набор вкладок волны 1 на мобайле — Чат / О эфире; десктоп-панель — только чат). Зеркалит то, как 005 отсрочила вкладки Записи / Сертификаты в `my-events`.
- **Автоначисление НМО (90 мин + 2 подтверждения) и авто-отчёт спонсору «Отчёт партнёра V2»** — волна 2. 006 только **фиксирует** данные присутствия, которые эти вертикали потребят; отчёт спонсору волны 1 — ручная выгрузка из них.
- **Артефакт сертификата** (баллы НМО vs скачиваемый PDF) — решение владельца волны 2; вне scope здесь (открытый вопрос PRD, оставлен на волну 2).
- **Точная форма отчёта партнёра V2** (колонки/джойны) — волна 2; ручной выгрузке волны 1 нужны только сырые минуты по каждому врачу (открытый вопрос PRD, волна 2).
- **SDN Player** — **не** в enum провайдера волны 1. Расширение закрытого списка сверх `rutube | youtube` позже — аддитивная **миграция**, а не форма, которую 006 пред-строит (решение владельца 2026-07-06).
- **Видеотранскодинг, собственный медиасервер, DRM / подписанное воспроизведение, телеметрия уровня плеера** — вне embed-рамки эпика (EARS-9).
- **Архив записей** — волна 2.
- **Публичная страница мероприятия + листинг (004), write регистрации + «мои события» (005) и флоу auth (003)** — потребляются, не принадлежат здесь.

## Ограничения (Constraints)

- **Гейт серверный, не UI-стена.** Допуск (аутентифицирован ∧ зарегистрирован ∧ `live`) принуждается бэкендом до отдачи любого содержимого комнаты — конфига плеера, credential чата или приёма heartbeat'а. Прямой URL комнаты, пересланная ссылка или сконструированный запрос никогда не рендерят плеер и не впускаются; легаси-паттерн мягкой стены поверх плеера запрещён (приёмка PRD; recon §6).
- **Регистрация — основа допуска, переиспользуемая, не переизобретаемая.** Проверка «зарегистрирован» читает `EventRoster` из 005 для `(врач, мероприятие)`; 006 добавляет **ни одного** примитива регистрации и не создаёт регистрацию. Незарегистрированного врача ведут в 005, а не впускают (EARS-1, EARS-6).
- **Выбор плеера — явный enum, никогда не URL-sniffing.** Провайдер плеера — закрытый список `rutube | youtube`, хранимый в стрим-конфиге мероприятия (авторство в 007); комната инстанцирует embed из значения enum, никогда не инспектируя строку URL. Неизвестный/отсутствующий провайдер — правдивое состояние «стрим недоступен», а не угаданный embed (решение владельца 2026-07-06; легаси-ошибка URL-sniffing, recon §5).
- **Каденс heartbeat N — server-side config, дефолт 60 с.** Арифметика минут присутствия **параметризована по N** — подтверждённый оператором иной каденс меняет **config**, а не спеку или код. Интервал постинга клиента управляется тем же значением с сервера (решение владельца 2026-07-06; легаси-свидетельство 60 с, recon §10-3).
- **Присутствие трекается только на видимой, активной вкладке.** Клиентский цикл heartbeat гейтирован Page Visibility API: пока вкладка комнаты в фоне (`document.hidden`), клиент **не** шлёт сигналов, поэтому минуты фоновой вкладки не идут в спонсорский отчёт; цикл возобновляется, когда вкладка снова становится видимой. Это **клиентский гейт фиксации** — сервер по-прежнему отклоняет любой сигнал от негейтированного вызывающего или закрытой комнаты. Коалесценция вкладок (EARS-5) неизменна: две _видимые_ сессии в одном интервале по-прежнему считаются один раз. Пороги НМО и интерактивные подтверждения присутствия остаются волной 2 (решение владельца 2026-07-07).
- **Присутствие server-authoritative и долговечно.** Сигналы принимаются только от аутентифицированного гейтированного врача и дописываются в долговечную таблицу Postgres (никогда не клиентски-доверенный счётчик, никогда не через exposed-service-key, которым легаси пинговал клиента, recon §6). Presence в Centrifugo эфемерно и **не** является долговечной записью — минуты идут из append-таблицы (adopt-vs-build брифа).
- **Параллельные вкладки не должны завышать минуты.** Сигналы из параллельных сессий врача по одному мероприятию коалесцируются в один таймлайн присутствия; две вкладки, открытые в одну минуту, считаются одной минутой присутствия (приёмка PRD).
- **Heartbeat'ы и сообщения чата останавливаются при закрытии комнаты.** Как только мероприятие выходит из `live` (комната закрыта 007), сервер отклоняет дальнейшие сигналы и сообщения по мероприятию; минуты вычисляются за окно, пока комната была открыта (EARS-7).
- **Только embed-рамка.** Никакого транскодинга, медиасервера, DRM/подписанного воспроизведения, рехостинга или телеметрии плеера — комната компонует сконфигурированный iframe/плеер вокруг внешнего стрима спонсора и только (scope эпика; buy-опция Mediator.cloud отклонена).
- **Чат едет на Centrifugo, уже в стеке.** Никакого нового решения по realtime-инфраструктуре — Centrifugo провижнен на дев-стенде и в engineering-readiness-спеке; 006 добавляет канал комнаты + gate-scoped connection-токен, а не новый транспорт (adopt-vs-build брифа; dev-stand.md).
- **Никакого хардкода origin / URL / ссылки на стрим.** Origin портала, эндпоинты API, URL Centrifugo и ссылка на стрим читаются из конфигурации / записи мероприятия, никогда не хардкодятся в коде или спеке (AGENTS.md §9; dev-stand.md).
- **Каноничный часовой пояс.** Любая абсолютная дата/время в комнате (расписание «О эфире») представлены в `Europe/Moscow` с пометкой **МСК**, вычислены из каноничного момента read-модели, и не должны съезжать в локальный пояс зрителя (зеркалит 004 EARS-12 / 005 EARS-11). Индикатор эфирного времени («В эфире · N мин») выводится из **фактического момента выхода в эфир** (`live_at`, проставляется 007 `OpenRoom` на переходе `published → live`) — **никогда** из запланированного `startsAt` (эфир, начавшийся с опозданием, не должен показывать завышенное время); у legacy-строки `live` без `live_at` плашка рендерится без суффикса минут (правдиво, без досчёта из расписания).
- **Канвас — спецификация точности (ADR-0013 / ADR-0014 §4).** Комната строится из токенов `@ds/design-system` по геометрии завендоренного `webinar-room.dc.html` (границы 2px, тень `6px 6px 0` на плеере + чат-панели, пульсирующий live-бейдж, десктоп-сетка `1fr 400px`, мобильный табовый макет). Произвольные значения Tailwind заблокированы линтером; где проза и канвас расходятся, побеждает канвас.
- **Стек** (ADR-0002 / ADR-0004): read конфига комнаты + команды чата и heartbeat — NestJS + `nestjs-zod`, SSOT схем в `packages/schemas/`, URI-версионирование `/v1/...`, Vitest + supertest; портал — Next.js 15. Зависящие от сервиса тесты `skipIf` при отсутствии их env-зависимости (`DATABASE_URL`, `IDP_ISSUER`, `CENTRIFUGO_URL`), чтобы не краснить общий CI unit job.

## Prior decisions (предшествующие решения)

- **ADR-0014** Жизненный цикл продуктового дизайна — PRD (`006-product.md`, US-1…US-5) — **источник** этого триплета; каждый EARS несёт `realizes: US-N` (§2). Завендоренный канвас `webinar-room.dc.html` — _источник_, репозиторий держит _собранный артефакт_, верифицируемый по нему (§3/§4).
- **ADR-0001 §2** Identity / Auth / RBAC — read конфига комнаты, подключение/публикация в чат и heartbeat-эндпоинт классифицированы `access: authenticated`, `required_roles: doctor_guest`, `auth_check: policy`. Гейт «зарегистрирован и live» — **policy**-оценка (зарегистрирован против списка 005 ∧ мероприятие `live`) — первые эндпоинты с auth_check `policy` в домене вебинаров (004 добавила `public`-reads, 005 — `fast-path` `doctor_guest` writes/reads). Новый auth-примитив не добавляется; сессия — отгруженная BFF-сессия 003.
- **ADR-0002 §3** Backend Core Stack — `nestjs-zod`, URI-версионирование `/v1/...`, Vitest + supertest, SSOT DTO `RoomConfig`, сообщения чата и heartbeat в `packages/schemas/`.
- **ADR-0003 §3/§6** Data Layer — сигналы присутствия живут в **append-only** таблице Postgres через Drizzle (без in-place update, без клиентски-доверенного счётчика); факты открытия/закрытия комнаты и первого входа добавляются в `audit_ledger`. Append-таблица — долговечная запись за спонсорскими минутами; presence Centrifugo эфемерно и не используется для отчёта.
- **ADR-0004** Frontend Stack — портал — Next.js 15; комната — поверхность портала. Режиссёрская консоль (открытие/закрытие, стрим-конфиг, 007) — Refine/admin, вне scope здесь.
- **ADR-0013** Design token SoT — нео-брутал-язык отгружается как токены/компоненты `@ds/design-system`; 006 строит комнату из токенов по точности завендоренного `webinar-room.dc.html`. Embed-рамка плеера и чат-панель — net-new юниты → сначала прогоняется гейт `build-ui-from-design-system` (chat-примитивы shadcn / react-player + тонкий iframe Rutube по adopt-vs-build эпика), фиксируется в PR.
- **ADR-0006 §4** Documentation & SSOT — структура триплета фича-спеки + плоская нумерация EARS.

## Модель событий (Event Model)

Фича 006 — **вертикаль комнаты**: `doctor_guest`-аутентифицированный, гейтированный «зарегистрирован и live» read (`RoomConfig`) плюс две гейтированные команды (`PostChatMessage`, `RecordPresenceHeartbeat`) с долговечной append-only-таблицей присутствия. Она **читает** `EventRoster` из 005 (основа допуска) и `EventLifecycleState` + стрим-конфиг из 004/007; она **производит** сигналы присутствия, из которых берётся спонсорский отчёт. Она владеет **ни одним** auth-примитивом (переиспользует 003), **ни** регистрацией (читает 005), **ни** авторством мероприятий / открытием-закрытием комнаты (читает 007).

### Команды (обрабатываются `apps/api`, аутентифицированы `doctor_guest` + `policy`-гейт «зарегистрирован и live»)

- `PostChatMessage(idOrSlug, text)` — постит сообщение в канал комнаты; бэкенд авторизует гейт, затем публикует в Centrifugo. Отклоняется, если комната не открыта (мероприятие не `live`) или вызывающий не гейтирован.
- `RecordPresenceHeartbeat(idOrSlug)` — дописывает один сигнал присутствия для `(врач, мероприятие, момент)`. Отклоняется, если комната не открыта или вызывающий не гейтирован. Идемпотентна в пределах интервала (параллельные вкладки коалесцируются; EARS-5).

### События

| Событие                                                    | Владелец                      | Заметки                                                                                                                                                                           |
| ---------------------------------------------------------- | ----------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `PresenceHeartbeatRecorded`                                | `apps/api`                    | Одна append-only-строка на принятый сигнал `(врач, мероприятие, момент)`. Долговечная основа минут присутствия по каждому врачу (US-3, US-4). Отклоняется после закрытия комнаты. |
| `ChatMessagePosted`                                        | `apps/api` (через Centrifugo) | Сообщение гейтированного врача разлетается по каналу комнаты в реальном времени (US-2). Не долговечная запись присутствия.                                                        |
| `EventWentLive` / `EventEnded` (комната открыта / закрыта) | 007 (потребляются на чтение)  | 006 читает итоговый `EventLifecycleState`: `live` ⇒ комната открыта (сигналы + сообщения принимаются); выход из `live` ⇒ комната закрыта (сигналы + сообщения отклоняются).       |
| `DoctorRegisteredForEvent`                                 | 005 (потребляется на чтение)  | 006 читает итоговый `EventRoster`, чтобы решить допуск для `(врач, мероприятие)`. 006 не создаёт регистрацию.                                                                     |

### Read-модели

- **`RoomConfig`** — per гейтированный вызывающий, per live-мероприятие: `{ provider ∈ {rutube, youtube}, embedRef, chatToken (Centrifugo, канал комнаты), heartbeatIntervalSeconds (= N) }`. Отдаётся **только** аутентифицированному, зарегистрированному врачу для `live`-мероприятия; никогда гостю, незарегистрированному врачу или для не-`live` мероприятия.
- **`EventPresence`** — append-only-сигналы по мероприятию; выведенные **минуты присутствия по каждому врачу** (параметризованы по N, коалесцированы по параллельным вкладкам) — спонсорский deliverable (ручная выгрузка в волне 1). Никогда не выставляется на публичную поверхность; PII зарегистрированных не на публичном эндпоинте (recon §6).
- **`EventRoster`** (принадлежит 005, **потребляется** здесь) — множество актуальных регистраций, основа допуска; **`EventLifecycleState`** + **стрим-конфиг** (принадлежат 007, отдаются 004, **потребляются** здесь) — проверка live + enum провайдера/ссылка на embed.

### Политики

- **На read `RoomConfig` / `PostChatMessage` / `RecordPresenceHeartbeat`** → впустить тогда и только тогда, когда `аутентифицирован ∧ зарегистрирован (список 005) ∧ мероприятие live`; иначе отклонить на сервере и зароутить (EARS-6). Никакая мягкая UI-стена не отдаёт содержимое негейтированному вызывающему.
- **На `RecordPresenceHeartbeat`** → дописать один сигнал; параллельные вкладки для одного `(врач, мероприятие)` коалесцируются в один таймлайн присутствия (без завышения минут; EARS-5). Сигналы отклоняются, как только комната закрыта (EARS-7).
- **На выходе мероприятия из `live`** (комната закрыта, 007) → отклонять дальнейшие сигналы и сообщения по мероприятию; минуты вычисляются за открытое окно (EARS-7).

## EARS-требования

> **Соглашение о нумерации:** плоское (`EARS-1`, `EARS-2`, …) по ADR-0006 §4. EARS-1…7 — функциональные обработчики (каждый становится дочерним Issue); EARS-8…11 — кросс-срезовые повсеместные / нежелательного-поведения требования, применяемые по всей поверхности. Каждое положение несёт бэклинк `realizes: US-N` к истории PRD, которую формализует (ADR-0014 §2).

**Допуск в комнату и композиция**

- **EARS-1** _(realizes: US-1, US-5)_ — Когда вызывающий запрашивает содержимое комнаты для мероприятия (конфиг плеера, credential-подключение к чату или приём heartbeat'а), система должна отдать его **только если** вызывающий **аутентифицирован И зарегистрирован на это мероприятие (список 005) И мероприятие `live`**, выдав серверный грант `RoomAccess`; прямой URL комнаты, пересланная ссылка или сконструированный запрос, не выполняющие любое из трёх условий, должны быть **отклонены на сервере** — нет мягкой UI-стены, которая рендерит плеер, чат или принимает сигнал для негейтированного вызывающего.
- **EARS-2** _(realizes: US-1)_ — Где существует грант `RoomAccess`, система должна отрендерить комнату по композиции `webinar-room.dc.html` (десктоп `1fr 400px` плеер + чат-панель; мобайл — full-bleed плеер + вкладки Чат / О эфире) и **инстанцировать embed-плеер из явного enum провайдера** стрим-конфига мероприятия — закрытого списка `rutube | youtube` — **никогда** не инспектируя и не сниффя URL стрима; неизвестный или отсутствующий провайдер должен давать правдивое состояние «стрим недоступен», а не угаданный embed.

**Живой чат**

- **EARS-3** _(realizes: US-2)_ — Где комната открыта, система должна давать гейтированному врачу **читать живой чат и постить сообщения через Centrifugo** (канал комнаты по id мероприятия), разлетая каждое отправленное сообщение всем участникам **в реальном времени без перезагрузки**; чат сессии доступен, пока комната открыта, а сообщение постится только через гейтированную команду (ни один клиент не может публиковать в канал комнаты без выданного сервером gate-scoped-токена).

**Фиксация присутствия**

- **EARS-4** _(realizes: US-3)_ — Пока гейтированный врач присутствует в live-комнате **и вкладка комнаты — видимая, активная вкладка** (Page Visibility API — `document.hidden` равен false), клиент должен постить **аутентифицированный heartbeat каждые N секунд** — N это **server-side config, дефолт 60 с**, доставляемый клиенту в `RoomConfig` — и система должна дописывать каждый принятый сигнал в **долговечную append-only-таблицу Postgres** с `(врач, мероприятие, момент)`; **пока вкладка комнаты в фоне (`document.hidden` равен true), клиент не должен слать heartbeat'ы — минуты фоновой вкладки не идут в спонсорский отчёт — и цикл heartbeat должен возобновиться, когда вкладка снова становится видимой**. Присутствие фиксируется с первой минуты **без** действия врача «докажи, что ты здесь».
- **EARS-5** _(realizes: US-3, US-4)_ — Append-only-сигналы должны давать **фактические минуты присутствия по каждому врачу** для мероприятия, вычисленные из меток времени сигналов **параметризованно по N** (подтверждённый оператором иной каденс меняет config, а не спеку или код); **параллельные вкладки одного врача не должны завышать** минуты этого врача (сигналы из параллельных сессий коалесцируются в один таймлайн присутствия); а зафиксированных данных должно быть **достаточно для спонсорского отчёта первого вебинара через ручную выгрузку** — без UI отчётов в волне 1.

**Ветки доступа и закрытие комнаты**

- **EARS-6** _(realizes: US-1, US-5)_ — Когда вызывающий достигает комнаты, но **недопустим**, система должна зароутить его **правдиво**, а не показать мягкую стену: **неаутентифицированного** посетителя проводят через флоу auth (003) и переоценивают; **аутентифицированного, но незарегистрированного** врача ведут зарегистрироваться (005) и впускают по успеху; мероприятие, которое **не `live`** (ещё не открыто или уже закрыто), показывает правдивое состояние жизненного цикла (состояния страницы 004) без смотрибельной комнаты.
- **EARS-7** _(realizes: US-3, US-4)_ — Когда мероприятие выходит из `live` (режиссёр закрывает комнату, фича 007), система должна **перестать принимать heartbeat'ы и сообщения чата** по мероприятию — поздний сигнал или сообщение отклоняются на сервере — и комната должна деградировать в правдивое ended-состояние; минуты присутствия по каждому врачу вычисляются за сигналы, зафиксированные **пока комната была открыта**.

**Кросс-срезовое (повсеместное / нежелательное поведение)**

- **EARS-8** _(realizes: US-5)_ — Read `RoomConfig`, эндпоинт подключения/публикации в чат и heartbeat-эндпоинт должны быть классифицированы **`access: authenticated`, `required_roles: doctor_guest`, `auth_check: policy`** в матрице endpoint-authz (ADR-0001 §2) — гейт «зарегистрирован и live» это policy-оценка сверх role fast-path. **Ни один** эндпоинт не отдаёт содержимое комнаты, credential чата и не принимает сигнал для неаутентифицированного или незарегистрированного вызывающего; **ни один** врач никогда не получает чат-идентичность, данные присутствия или список другого врача; а данные присутствия по каждому врачу никогда не выставляются на публичную поверхность.
- **EARS-9** _(realizes: US-1)_ — Система должна встраивать внешний стрим как **рамку и только**: она **не** должна транскодировать, рехостить, проксировать, DRM-подписывать или записывать стрим и **не** должна эмитить телеметрию уровня плеера — комната компонует сконфигурированный iframe/плеер вокруг стрима спонсора и не более (граница embed-рамки эпика; buy-опция Mediator.cloud отклонена).
- **EARS-10** _(realizes: US-1, US-2, US-3)_ — Поверхности комнаты 006 должны рендериться на **русском** с **отсутствием хардкод-строк** — весь копирайт (live-бейдж, chrome комнаты, плейсхолдер/лейблы/пустое состояние чата, указания веток доступа и состояния «стрим недоступен»/ended) берётся из типизированного каталога сообщений из 003 (EARS-21) и 004 / 005; любая **абсолютная** дата/время (расписание «О эфире») представлены в `Europe/Moscow` с пометкой **МСК**, а индикатор эфирного времени выводится из **фактического момента выхода в эфир** (`live_at`), а не из запланированного `startsAt`.
- **EARS-11** _(realizes: US-1, US-2)_ — Комната должна быть построена из **токенов `@ds/design-system`** по геометрии завендоренного `webinar-room.dc.html` (границы 2px, тень `6px 6px 0` на плеере + чат-панели, пульсирующий live-бейдж, десктоп-сетка `1fr 400px`, мобильный табовый макет), корректно рендерясь на **обоих брейкпоинтах × обеих темах**; произвольные значения Tailwind заблокированы линтером (ADR-0013), а любой bespoke-элемент сверх инвентаря DS (embed-рамка плеера, чат-панель) сначала прогоняет гейт `build-ui-from-design-system` (chat-примитивы shadcn / react-player + тонкий iframe Rutube по adopt-vs-build эпика) и фиксируется в PR.

## Инварианты

- Содержимое комнаты (конфиг плеера, credential чата, приём heartbeat'а) отдаётся **только** аутентифицированному, зарегистрированному врачу для `live`-мероприятия — ни один путь (прямой URL, пересланная ссылка, сконструированный запрос) не впускает негейтированного вызывающего; нет мягкой UI-стены (EARS-1, EARS-8).
- Провайдер плеера всегда один из **закрытого enum** `rutube | youtube`, читаемого из стрим-конфига мероприятия — никогда не выводится из URL; неизвестный/отсутствующий провайдер — правдивое состояние «стрим недоступен», никогда не угаданный embed (EARS-2).
- Клиентский цикл heartbeat **гейтирован видимостью** — фоновая вкладка комнаты (`document.hidden`) **не** шлёт сигналов, поэтому фиксируется только присутствие видимой, активной вкладки; цикл возобновляется при повторной видимости (EARS-4).
- Каждый сигнал присутствия **server-authoritative и долговечен** — дописан в Postgres для аутентифицированного гейтированного врача, никогда не клиентски-доверенный счётчик и никогда не через exposed-key (EARS-4).
- Параллельные вкладки для одного `(врач, мероприятие)` **никогда не завышают** минуты присутствия — параллельные сигналы коалесцируются в один таймлайн присутствия (EARS-5).
- Сигналы и сообщения чата принимаются **только пока комната открыта** (`live`); как только комната закрыта — отклоняются, а минуты вычисляются за открытое окно (EARS-7).
- Минуты присутствия **параметризованы по N** (server config, дефолт 60 с) — иной подтверждённый каденс меняет config, а не спеку или код (EARS-4, EARS-5).
- Данные присутствия по каждому врачу и список **никогда** не на публичной поверхности — PII зарегистрированных не утекает на публичный эндпоинт (EARS-8; recon §6).
- Комната **встраивает только** — она никогда не транскодирует, не рехостит, не DRM-подписывает, не записывает и не телеметрирует стрим (EARS-9).
- Любая абсолютная дата/время в комнате — в `Europe/Moscow` с пометкой МСК, выведены из одного каноничного момента (EARS-10).

## Зависимости и секвенирование

- **MVP-критичный дедлайн — 2026-07-17.** 006 — поверхность, на которой реальные врачи смотрят живой вебинар; она должна работать end-to-end на новой платформе к дате эфира, с **нулём жалоб «не смог подключиться» по вине платформы** (приёмка PRD; success-метрика брифа). Секвенирование ниже держит 006 завершаемой как собственную вертикаль на seed'ах + реальных 003/004/005, чтобы дедлайн не ждал 007.
- **Фундамент auth (фича 003) — отгружен.** Гейт комнаты переиспользует BFF-сессию 003 как аутентифицированного принципала; неаутентифицированного посетителя проводят через отгруженный флоу 003 (EARS-6). 006 не добавляет auth-примитивов.
- **Публичная страница мероприятия + путь входа (фича 004).** В комнату входят со страницы мероприятия 004 / пути входа; не-`live` мероприятие откатывается на состояние жизненного цикла 004 (EARS-6). 006 секвенирована **после** того, как 004 на `main` (путь входа существует, чтобы на нём строить); до тех пор комната E2E-прогоняется с seed-страницы live-мероприятия.
- **Запись регистрации + `EventRoster` (фича 005).** Допуск (EARS-1) читает `EventRoster` из 005 для `(врач, мероприятие)`; незарегистрированного врача роутят в 005 (EARS-6). 006 секвенирована **после** того, как 005 на `main` (список — основа допуска); это блокирующая связка 005↔006, которую вяжет `open-ears-issues` (его шаг 4). До тех пор допуск E2E-прогоняется на **seed-списке**.
- **Режиссёрское открытие/закрытие + стрим-конфиг (фича 007) — отслеживаемый шов.** Окно `live` комнаты (открытие/закрытие) и стрим-конфиг (enum провайдера + ссылка на embed) авторизуются/двигаются 007. Пока 007 не приземлится, 006 строится + E2E-прогоняется на **seed-мероприятиях live с seed-стрим-конфигом + seed-списком** в каждом состоянии (live / ended). Критерий «сделано против реальной зависимости» — _«комната открывается/закрывается через режиссёрские органы 007 и инстанцирует плеер из стрим-конфига, авторизованного 007, а не из seed'ов»_, несомый на блокирующей связке 006↔007 — ровно как 005 несла свою зависимость от 007. Ни один scaffold не подменяет 007 молча: шов — отслеживаемый Issue с этим явным критерием (AGENTS.md §6 F-22).
- **Centrifugo — уже в стеке.** Чат едет на Centrifugo, провижнен на дев-стенде и в engineering-readiness-спеке (adopt-vs-build брифа; `.claude/rules/dev-stand.md`). 006 добавляет канал комнаты + gate-scoped connection-токен, **никакого** нового решения по realtime-инфраструктуре. E2E чата гоняется против Centrifugo дев-стенда (`CENTRIFUGO_URL`), `skipIf` при отсутствии.
- **Дизайн-система (`@ds/design-system`, ADR-0013).** Нео-брутал-токены/примитивы на `main` (#510 → #512–515). Net-new юниты комнаты (embed-рамка плеера, чат-панель) сначала прогоняют гейт `build-ui-from-design-system` (chat-примитивы shadcn / react-player + тонкий iframe Rutube по adopt-vs-build эпика); любой bespoke-элемент фиксируется в PR.
- **Dev-стенд Postgres + Zitadel + Centrifugo.** e2e-тесты комнаты гоняются против Postgres дев-стенда (`DATABASE_URL`) + `idp` живого стенда + Centrifugo (`CENTRIFUGO_URL`) по `.claude/rules/dev-stand.md`; эндпоинты/порты читаются из `.env.local`.
