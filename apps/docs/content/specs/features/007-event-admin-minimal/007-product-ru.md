> Эпик: [«Вебинары» — продуктовый бриф](../../product/webinars/brief-ru.md) · Волна 1, утверждённый вариант A «тонкая вертикаль». Админ-поверхности живут в приложении `admin` (Refine, ADR-0004).

## Суть фичи

Минимальный инструментарий оператора/режиссёра, делающий возможными остальные три фичи: создать и отредактировать мероприятие (название, дата/время МСК, описание, спикеры текстом/ссылками, специальности, PDF программы, спонсор), сконфигурировать ссылку на стрим с явным провайдером, открыть и закрыть живую комнату в день эфира и провести мероприятие по жизненному циклу. Жизненный цикл — **единая state machine** — `draft → published → live → ended → archived`, — заменяющая легаси-россыпь булевых флагов (`draft` / `published?` / `archive` / `visible_in_rg` / …), из-за которой вопрос «видно ли это мероприятие?» был неоднозначным. Это сознательно **не** легаси-редактор на 11 вкладок и не полный режиссёрский пульт: без авторинга виджетов, без конструктора программы (им никто не пользовался — операторы загружают финальный PDF), без UI отчётов.

## Пользовательские истории

- **US-1** — Как **оператор**, я создаю мероприятие за ≥1 месяц с полями, нужными публичной странице — название, дата/время (МСК), описание, спикеры (текстом или ссылками), целевые специальности, PDF программы, спонсор, — и держу его в `draft`, пока оно не готово к анонсу.
- **US-2** — Как **оператор**, я редактирую опубликованное мероприятие по мере уточнения деталей (программа «часто меняется» — я загружаю финальную версию PDF), и публичная страница отражает правку.
- **US-3** — Как **оператор/режиссёр**, в день эфира я вставляю ссылку на стрим и явно выбираю провайдера — комната точно знает, какой плеер встраивать, без угадывания по URL.
- **US-4** — Как **режиссёр**, я открываю живую комнату на старте эфира и закрываю по его окончании — именно эти два действия впускают зрителей и ограничивают фиксацию присутствия.
- **US-5** — Как **оператор**, я веду мероприятие по жизненному циклу — публикую, а после эфира довожу через `ended` до `archived`, — и система предлагает только те переходы, которые осмысленны из текущего состояния.

## Флоу

**Happy path — полная дуга одного вебинара (US-1 → US-5):**

1. За ≥1 месяц: оператор создаёт мероприятие в админке, заполняет базовые поля, загружает PDF программы → `draft`.
2. Оператор публикует → `published`: мероприятие появляется в листинге portal, его страница становится публично читаемой (фича 004); открывается регистрация (фича 005).
3. День эфира: режиссёр вставляет URL стрима + выбирает провайдера (например, `rutube`), проверяет конфиг и **открывает комнату** → `live`: зарегистрированные врачи допускаются (фича 006), фиксация присутствия работает.
4. Эфир окончен: режиссёр **закрывает комнату** → `ended`: допуск в комнату и приём heartbeat'ов прекращаются; данные присутствия полны для передачи спонсору.
5. Позже: оператор архивирует → `archived`: мероприятие уходит с публичных поверхностей.

**Ключевые ветки:**

- PDF программы заменён после публикации → страница мероприятия отдаёт новый файл (US-2).
- Ссылка на стрим исправлена в состоянии `published` (вставлен неверный URL) → конфиг комнаты обновляется до/при открытии.
- Невалидные скачки жизненного цикла (например, `draft → live`, повторное открытие `archived`) не предлагаются.

## Продуктовые критерии приёмки

- Оператор может создать и отредактировать мероприятие в приложении `admin` с полями: название, дата + время явно в **МСК**, описание, спикеры (свободный текст и/или ссылки), целевые специальности, **PDF** программы, спонсор/партнёр.
- Жизненный цикл мероприятия — одно поле с одним закрытым набором состояний (`draft → published → live → ended → archived`); UI предлагает только валидные переходы из текущего состояния — никаких булевых флагов на сверку.
- Стрим-конфигурация = URL + **явный выбор провайдера из закрытого enum**; комната (фича 006) потребляет ровно этот конфиг.
- «Открыть комнату» / «закрыть комнату» — явные действия режиссёра, привязанные к переходам `live` / `ended`; закрытие останавливает допуск и ограничивает фиксацию присутствия.
- То, что админка показывает как состояние мероприятия, — ровно то, что отражают поверхности portal (видимость в листинге, состояние страницы, доступ в комнату): один источник истины, без дрейфа.
- Минимальный путь — создать → опубликовать → сконфигурировать стрим → открыть → закрыть — проходим одним оператором/режиссёром для вебинара 2026-07-17 без вмешательства разработчика.

## Вне scope

- Авторинг виджетов / «титровальных объектов», пульт опросов, модерация вопросов лектору (волна 2).
- Конструктор программы (в легаси был; никто не пользовался — реальный workflow это загрузка финального PDF).
- UI отчётности / кнопки выгрузки отчётов (отчёт спонсору первого вебинара — ручная выгрузка из данных присутствия — данных фичи 006).
- Конфигурация НМО (коды, баллы, аккредитация) — волна 2.
- Управление справочником спикеров, per-event карточки спикеров с рейтингами, уровни партнёров (сверх простой ссылки на спонсора).
- Иерархия админ-ролей / списки в скоупе менеджера (легаси-RLS-ветки); волна 1 исходит из малой доверенной группы операторов.
- Конгресс-блоки лендинга, офлайн-мероприятия, платные флоу (вне эпика).

## Открытые вопросы

- Глубина представления спикеров в волне 1: свободного текста всегда достаточно для первого вебинара, но когда ссылки на реальные записи пользователей/спикеров становятся обязательными? (владелец)
- `ended → archived` — ручное действие оператора (как в легаси) или по времени? (владелец)
- Кто именно держит админ-доступ в волне 1 (оператор + режиссёр как одна доверенная группа или раздельные роли)? (владелец)
