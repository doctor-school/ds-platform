# 007 — Минимальная админка мероприятия: создание/редактирование, стрим-конфиг, управление комнатой, жизненный цикл (Требования)

> Эпик: [«Вебинары» — продуктовый бриф](../../product/webinars/brief-ru.md) · Волна 1, утверждённый вариант A «тонкая вертикаль». Источник PRD: [`007-product-ru.md`](./007-product-ru.md) (US-1…US-5). **Канваса-мокапа для админ-поверхности не существует (известный Stage-A gap, PRD «Утверждённый мокап»)** — волна 1 строится на **штатном UI Refine** (ADR-0004); чекпоинт владельца (оставить штатный vs заказать админ-канвас) зафиксирован ниже.

## Итоги (Outcomes)

- **Оператор** создаёт мероприятие за ≥1 месяц в приложении `admin` с набором полей, нужным публичной странице (004) — название, школа/серия, **дата + время явно в МСК** (хранится один каноничный момент), описание, спикеры, целевые специальности, **PDF** программы и ссылка на спонсора/партнёра — и держит его в `draft`, пока оно не готово к анонсу (US-1).
- **Оператор** редактирует опубликованное мероприятие по мере уточнения деталей — в частности **заменяет PDF программы** (он «часто меняется») — и публичная страница мероприятия (004) отражает правку, отдавая **текущий** файл (US-2).
- **Оператор/режиссёр** конфигурирует стрим, вставляя ссылку и **явно выбирая провайдера из закрытого enum** (`rutube | youtube`) плюс ссылку на embed, так что комната (006) инстанцирует ровно нужный плеер **без угадывания по URL**; неверный URL исправим в состоянии `published` (US-3).
- **Режиссёр** **открывает живую комнату** на старте эфира и **закрывает** по его окончании — эти два действия суть переходы `published → live` / `live → ended`, впускающие зарегистрированных зрителей (006) и ограничивающие фиксацию присутствия (US-4).
- **Оператор** ведёт мероприятие по **одному закрытому жизненному циклу** — `draft → published → live → ended → archived` — при этом админ-UI предлагает **только переходы, валидные из текущего состояния**, а сервер отклоняет любой невалидный скачок; **никаких булевых флагов на сверку** (US-5).
- **То, что админка показывает как состояние мероприятия, — ровно то, что отражают поверхности portal** — видимость в листинге (004), состояние страницы (004), доступность регистрации (005), доступ в комнату (006) — один источник истины `EventLifecycleState`, без дрейфа (US-1…US-5).
- **Минимальный путь — создать → опубликовать → сконфигурировать стрим → открыть → закрыть — проходим одним оператором/режиссёром** для вебинара 2026-07-17 без вмешательства разработчика (US-1…US-5, приёмка PRD).

## Lead authoring decisions (owner review pending)

PRD оставил три вопроса открытыми для владельца; на момент авторинга владелец был недоступен, поэтому каждый решён здесь как **lead decision (2026-07-06, owner review pending)** и может быть отменён на ревью спеки. Каждое также вынесено инлайн на тот EARS/Constraint, который оно формирует.

- **LD-1 — Представление спикеров = упорядоченные текстовые записи.** Волна 1 хранит спикеров как **упорядоченный список свободно-текстовых записей** (имя + регалии/строки), валидируемых только как текст. **Ссылки на реальные записи пользователей/спикеров отложены на волну 2**, в связке с отложенным управлением справочником спикеров — ссылке без справочника не на что указывать. Схема сохраняет поле как **расширяемый список записей**, чтобы вариант-ссылка волны 2 был аддитивной миграцией, а не переформовкой (LD-1; EARS-1, Scope → вне scope).
- **LD-2 — `ended → archived` — ручное действие оператора.** В волне 1 **нет планировщика и нет автоматизации по времени**: архивирование — явная команда оператора (легаси-паритет). **Авто-архив по времени** — именованный кандидат **волны 2** (LD-2; EARS-6, Scope → вне scope).
- **LD-3 — Админ-доступ = одна доверенная админ-группа (`platform_admin`).** Оператор и режиссёр — **одна платформенная роль** в волне 1 — единственная админ-группа `platform_admin` из модели IdP ADR-0001 §2. **Раздельные роли оператор/режиссёр и списки в скоупе менеджера (owner-of-record)** — волна 2 (LD-3; EARS-8, Constraints).

## Scope

**В scope:**

- **Создать мероприятие (→ `draft`)** — оператор создаёт мероприятие в админке с полями: название, школа/серия, **дата + время в МСК**, хранимые как один каноничный момент, длительность, описание, **спикеры как упорядоченные текстовые записи** (LD-1), целевые специальности, **PDF программы**, загружаемый в object storage, и ссылка на спонсора/партнёра. Новое мероприятие стартует в `draft` и не публично достижимо, пока не опубликовано (EARS-1).
- **Редактировать мероприятие + заменяемый PDF программы** — оператор редактирует поля мероприятия в любом до-архивном состоянии; **замена PDF программы после публикации** обновляет ссылку на хранимый объект, и публичная страница мероприятия (004) отдаёт **текущий** файл — предыдущий файл после замены не отдаётся (EARS-2).
- **Конфигурация стрима (явный enum провайдера + ссылка на embed)** — оператор/режиссёр задаёт стрим-конфиг из провайдера **закрытого enum `rutube | youtube`** плюс **ссылки на embed** (provider-scoped id стрима, никогда не URL для сниффинга); конфиг исправим в состоянии `published`, и комната (006) потребляет ровно этот конфиг (EARS-3).
- **Переход публикации (`draft → published`)** — оператор публикует мероприятие; при публикации оно становится публично достижимым в листинге + на странице 004, и **открывается регистрация** (005). Публикация отклоняется, если мероприятие не в `draft` (EARS-4).
- **Открыть комнату / закрыть комнату (`published → live` / `live → ended`)** — режиссёр **открывает комнату** (переход `published → live`), что впускает зарегистрированных врачей в комнату 006 и запускает фиксацию присутствия; и **закрывает комнату** (переход `live → ended`), что останавливает допуск и приём heartbeat/чата и ограничивает окно присутствия (EARS-5).
- **Переход архивирования (`ended → archived`)** — оператор **вручную** архивирует `ended`-мероприятие (LD-2); архивированное мероприятие уходит со всех публичных поверхностей (уведомление/выпадение из листинга 004). Без планировщика (EARS-6).
- **Единая state machine мероприятия** — одно поле `EventLifecycleState` с **закрытым набором переходов** `draft→published`, `published→live`, `live→ended`, `ended→archived`; админ-UI **предлагает только переходы, валидные из текущего состояния**, а сервер **отклоняет любой невалидный скачок** (например, `draft→live`, `published→ended`, повторное открытие `archived`, любой обратный ход) — принуждение серверное, не просто скрытие в UI (EARS-7).
- **Read-сторона админки + производимые read-модели** в `apps/api` — админ-read'ы оператора (`EventAdminList`, `EventAdminDetail`) плюс **производимые** проекции, потребляемые остальным эпиком (`EventLifecycleState`, `PublicEventPage` / `UpcomingBroadcastCard` для 004, стрим-конфиг для 006), все классифицированы **`authenticated` `platform_admin` `fast-path`** в матрице endpoint-authz (ADR-0001 §2) для админ-read'ов/write'ов (производимые публичные проекции сохраняют `public`-классификацию 004) (EARS-8).
- **Единый источник истины админ/portal** — тот же `EventLifecycleState`, который пишут команды 007, ведёт админ-отображение **и** публичные поверхности (004/005/006); нет второго флага видимости и нет дрейфа (EARS-9).
- **RU-копирайт админки из типизированного каталога сообщений** — без хардкод-строк на админ-поверхности, переиспользуя i18n-готовую структуру из 003 (EARS-21) и 004/005/006; любая абсолютная дата/время в `Europe/Moscow` с пометкой **МСК**, вычислена из каноничного момента (EARS-10).
- **Штатная Refine-поверхность админки** — построена на **штатном UI Refine** (ADR-0004) с гейтом adopt-before-bespoke; токен-дисциплина ADR-0013 применяется там, где появляется любой bespoke-стайлинг. **Канваса** для этой поверхности **нет** (Stage-A gap) → **нет** канвас-точных EARS; чекпоинт владельца (оставить штатный vs заказать админ-канвас) назван (EARS-11).

**Явно вне scope** (каждое — именованная отсрочка, не молчаливый дефолт):

- **Справочник спикеров / ссылки на реальные записи** — волна 1 хранит спикеров только как упорядоченные текстовые записи (LD-1); ссылки на реальные записи пользователей/спикеров, per-event карточки спикеров с рейтингами и управление справочником — **волна 2**.
- **Авто-архив по времени / по расписанию** — `ended → archived` в волне 1 это ручное действие оператора (LD-2); авто-архив по времени — кандидат волны 2. Планировщика в 007 нет.
- **Раздельные роли оператор/режиссёр + списки в скоупе менеджера** — волна 1 это одна доверенная группа `platform_admin` (LD-3); иерархия ролей (сплит оператор/режиссёр, скоупинг менеджера/owner-of-record, легаси-RLS-ветки) — волна 2.
- **Авторинг виджетов / «титровальных объектов», пульт опросов, модерация вопросов лектору** — волна 2 (комната 006 откладывает те же аффордансы).
- **Конструктор программы** — конструктор программы легаси-редактора на 11 вкладок **не** пересобирается; реальный workflow — загрузка одного финального PDF (PRD; recon).
- **UI отчётности / кнопки выгрузки отчётов** — отчёт спонсору первого вебинара — **ручная выгрузка** из данных присутствия 006; 007 не отгружает UI отчётов.
- **Конфигурация НМО** (коды, баллы, аккредитация) — волна 2.
- **Снятие с публикации / `published → draft`** — PRD **не** называет переход снятия с публикации, поэтому 007 его **не** изобретает; в закрытом наборе переходов нет обратного хода (EARS-7). Правка `published`-мероприятия — это редактирование (EARS-2) или исправление стрим-конфига (EARS-3), а не реверс состояния.
- **Регистрация, комната, сама публичная страница/листинг** — 007 **авторит** агрегат мероприятия + жизненный цикл + стрим-конфиг; write регистрации (005), допуск/чат/присутствие комнаты (006) и публичная страница/листинг (004) — **потребители**, принадлежащие тем фичам.
- **Конгресс-блоки лендинга, офлайн-мероприятия, платные флоу** — вне эпика.

## Ограничения (Constraints)

- **Одна state machine, один закрытый набор переходов — принуждаемый сервером.** Жизненный цикл — единый enum `EventLifecycleState` (`draft → published → live → ended → archived`), заменяющий легаси-россыпь булевых флагов (`draft` / `published?` / `archive` / `visible_in_rg` / …), из-за которой вопрос «видно ли это мероприятие?» был неоднозначным (recon §7d). **Единственные** легальные переходы — четыре форвардных хода; любой другой ход (обратный, скачок, повторное открытие archived) **отклоняется на сервере**, а админ-UI **предлагает только** переходы, валидные из текущего состояния. Скрытие невалидного перехода в UI необходимо, но **не** достаточно — сервер это авторитет (EARS-7).
- **Состояние админки — единственный источник истины.** Состояние, которое пишет 007, — это состояние, которое читают 004 (листинг/страница), 005 (гейтинг регистрации) и 006 (доступ в комнату); нет **второго** флага видимости на сверку и нет дрейфа (EARS-9, приёмка PRD).
- **Провайдер стрима — явный enum, никогда не URL-sniffing.** Стрим-конфиг — `{ provider ∈ {rutube, youtube}, embedRef }`; провайдер выбирается оператором явно, а ссылка на embed — provider-scoped id стрима — комната (006) никогда не выводит провайдера из строки URL (легаси-ошибка, recon §5). Расширение enum сверх `rutube | youtube` позже — аддитивная **миграция**, а не форма, которую 007 пред-строит (зеркалит решение владельца 006 от 2026-07-06).
- **PDF программы живёт в object storage и заменяем.** PDF программы загружается в object storage платформы (Timeweb Object Storage в prod; MinIO на дев-стенде, ADR-0003 §, `.claude/rules/dev-stand.md`) — никогда не коммитится в репозиторий и не встраивается в запись — и **заменяем после публикации**: замена обновляет хранимую ссылку, так что публичная страница 004 отдаёт текущий файл, а вытесненный файл больше не отдаётся (EARS-2, US-2).
- **Каноничное московское время.** Оператор вводит дату + время, понимаемые как **МСК**; система хранит **один каноничный момент**, и любая абсолютная дата/время, рендеримые на админ-поверхности (и производимые для 004/005/006), представлены в `Europe/Moscow` с пометкой **МСК**, никогда не съезжая в локальный пояс оператора (зеркалит 004 EARS-12 / 005 EARS-11 / 006 EARS-10).
- **Админ-доступ — одна доверенная группа (`platform_admin`), role fast-path.** Админ-команды и админ-read'ы требуют роли `platform_admin` (LD-3); в волне 1 проверка — **role fast-path** (`auth_check: fast-path`) — роли одной достаточно, потому что группа одна доверенная без per-event скоупинга владельца. Переходы жизненного цикла и write'ы стрим-конфига **никогда** не вызываемы `doctor_guest` или публичным вызывающим. Админ-мутации — **высокоставочный** tier (ADR-0001 §2.5/§8): админ-приложение Refine закрыто 2FA (ADR-0004 §3), а меняющие состояние команды идут по пути валидации через introspection; волна 1 не добавляет bespoke per-action step-up сверх этой позиции (EARS-8).
- **007 переиспользует отгруженный auth (003), не добавляет auth-примитивов.** Админ-принципал — сессия `platform_admin`, выданная отгруженным слоем 003/IdP; 007 не вводит нового credential, сессии или токен-пути (ADR-0001; Constraints 003).
- **Никакого хардкода origin / URL / storage-эндпоинта.** Origin админки, эндпоинты API, эндпоинт/бакет object storage и ссылка на стрим читаются из конфигурации / записи мероприятия, никогда не хардкодятся в коде или спеке (AGENTS.md §9; `.claude/rules/dev-stand.md`).
- **Штатный Refine, adopt-before-bespoke.** Админ-поверхность использует **штатные компоненты Refine** (ADR-0004); гейт `build-ui-from-design-system` прогоняется до любого bespoke-элемента, а токен-дисциплина ADR-0013 (без произвольного Tailwind) применяется там, где появляется кастомный стайлинг. **Админ-канваса нет** (Stage-A gap), поэтому — в отличие от 004/005/006 — 007 не несёт **никакого** канвас-точного требования; владелец решает на ближайшем чекпоинте Stage-A, оставить штатный или заказать канвас (EARS-11).
- **Стек** (ADR-0002 / ADR-0004): команды авторинга мероприятия, переходы жизненного цикла и админ/производимые read'ы — NestJS + `nestjs-zod`, SSOT схем в `packages/schemas/`, URI-версионирование `/v1/...`, Vitest + supertest; админ-приложение — Refine (клиентское) на NestJS API через кастомные data/auth/access-провайдеры. Зависящие от сервиса тесты `skipIf` при отсутствии их env-зависимости (`DATABASE_URL`, `IDP_ISSUER`, `S3_ENDPOINT`), чтобы не краснить общий CI unit job.

## Модель событий (Event Model)

Фича 007 — **вертикаль авторинга** — **write-сторона** агрегата вебинара и **производитель** состояния, потребляемого read-сторонними вертикалями. Она владеет агрегатом мероприятия, `stream_config`, ссылкой на PDF программы и единой state machine `EventLifecycleState`. Она **производит** `EventLifecycleState`, проекции 004 `PublicEventPage` / `UpcomingBroadcastCard` и стрим-конфиг 006; она **потребляет** только отгруженный auth 003 (принципал `platform_admin`) и object storage (PDF). Она владеет **ни** регистрацией (005), **ни** допуском/чатом/присутствием комнаты (006), **ни** публичным рендерингом (004).

### Команды (обрабатываются `apps/api`, аутентифицированы `platform_admin`, role fast-path)

- `CreateEvent(fields)` — создать мероприятие в `draft` с полным набором полей (название, школа, каноничный `startsAt` из МСК-ввода, длительность, описание, упорядоченные текстовые спикеры, специальности, ссылка на PDF программы, спонсор/партнёр). Эмитит `EventCreated`.
- `UpdateEvent(idOrSlug, fields)` — редактировать любое поле до-архивного мероприятия, **включая замену PDF программы** (новая загрузка вытесняет хранимую ссылку). Эмитит `EventUpdated`.
- `ConfigureStream(idOrSlug, { provider ∈ {rutube, youtube}, embedRef })` — задать/исправить стрим-конфиг; исправим в `published`. Эмитит `StreamConfigured`.
- `PublishEvent(idOrSlug)` — `draft → published`; мероприятие становится публично достижимым (004), регистрация открывается (005). Отклоняется, если не `draft`. Эмитит `EventPublished`.
- `OpenRoom(idOrSlug)` — `published → live`; впускает зарегистрированных врачей в комнату 006 и запускает фиксацию присутствия. Отклоняется, если не `published`. Эмитит `EventWentLive`.
- `CloseRoom(idOrSlug)` — `live → ended`; останавливает допуск в комнату + приём heartbeat/чата и ограничивает окно присутствия (006 EARS-7). Отклоняется, если не `live`. Эмитит `EventEnded`.
- `ArchiveEvent(idOrSlug)` — `ended → archived`; мероприятие уходит со всех публичных поверхностей. **Вручную** (LD-2). Отклоняется, если не `ended`. Эмитит `EventArchived`.

Каждая команда перехода валидирует **текущее** состояние и отклоняет невалидный скачок (EARS-7); поверхность команд не предлагает перехода, который текущее состояние не разрешает.

### События

| Событие                           | Владелец (007) | Заметки                                                                                                                                                                    |
| --------------------------------- | -------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `EventCreated` / `EventUpdated`   | `apps/api`     | Авторенный/отредактированный агрегат (вкл. ссылку на PDF программы). Замена PDF вытесняет хранимую ссылку; 004 отдаёт текущий файл (US-1, US-2).                           |
| `StreamConfigured`                | `apps/api`     | `{ provider, embedRef }`, из которого комната 006 инстанцирует плеер; исправим в `published` (US-3).                                                                       |
| `EventPublished`                  | `apps/api`     | `draft → published`. Потребляется на чтение 004 (публичная достижимость + листинг) и 005 (открывается регистрация). Одна строка `audit_ledger`.                            |
| `EventWentLive` (комната открыта) | `apps/api`     | `published → live`. Потребляется 006: комната открывается (старт допуска + фиксации присутствия) и 004 (сигнал «в эфире»). Одна строка `audit_ledger`.                     |
| `EventEnded` (комната закрыта)    | `apps/api`     | `live → ended`. Потребляется 006: комната закрывается (сигналы + сообщения отклоняются, окно присутствия ограничено) и 004 (ended-состояние). Одна строка `audit_ledger`.  |
| `EventArchived`                   | `apps/api`     | `ended → archived` (вручную, LD-2). Потребляется 004: мероприятие уходит из публичных листингов; прямая ссылка рендерит уведомление об архиве. Одна строка `audit_ledger`. |

### Read-модели

- **`EventAdminList`** — список **всех** мероприятий оператора вне зависимости от состояния (`draft`…`archived`), каждое с текущим `EventLifecycleState`, датой/временем эфира (МСК) и полнотой стрим-конфига — индекс админ-приложения. Аутентифицирован `platform_admin`; никогда не публичен.
- **`EventAdminDetail`** — полный редактируемый агрегат одного мероприятия (все поля + стрим-конфиг + ссылка на PDF программы + валидные следующие переходы из текущего состояния). Аутентифицирован `platform_admin`.
- **`EventLifecycleState`** (производится) — единый enum `draft | published | live | ended | archived`; источник истины, читаемый каждой поверхностью 004/005/006.
- **`PublicEventPage` / `UpcomingBroadcastCard`** (производятся; **отдаются** 004) — publish-safe проекция авторенного агрегата (вкл. текущий `programPdfUrl`); 004 владеет публичными read'ами/рендерингом, 007 владеет авторенными данными за ними.
- **стрим-конфиг** (производится; **потребляется** 006) — `{ provider ∈ {rutube, youtube}, embedRef }`; комната инстанцирует плеер из него, никогда не из URL.

### Политики

- **На любой команде перехода** → валидировать текущее состояние и применить **только** легальный форвардный ход (`draft→published→live→ended→archived`); отклонить невалидный скачок на сервере, предлагая в админ-UI только переходы, которые разрешает текущее состояние (EARS-7).
- **На `PublishEvent`** → мероприятие становится публично достижимым (004), регистрация открывается (005) — единственный сигнал видимости, без булевого флага (EARS-4, EARS-9).
- **На `OpenRoom` / `CloseRoom`** → окно открытия комнаты 006 — ровно состояние `live`; открытие запускает допуск + фиксацию присутствия, закрытие останавливает их и ограничивает окно присутствия (EARS-5; 006 EARS-7).
- **На `UpdateEvent` с новым PDF программы** → хранимая ссылка заменяется, и страница 004 отдаёт текущий файл; вытесненный файл больше не отдаётся (EARS-2).
- **На каждой команде/read** → требовать `platform_admin`; никогда не отдавать `doctor_guest` или публичному вызывающему; добавлять одну строку `audit_ledger` на переход жизненного цикла (EARS-8; ADR-0003 §6).

## EARS-требования

> **Соглашение о нумерации:** плоское (`EARS-1`, `EARS-2`, …) по ADR-0006 §4. EARS-1…7 — функциональные обработчики (каждый становится дочерним Issue); EARS-8…11 — кросс-срезовые повсеместные / нежелательного-поведения требования, применяемые по всей поверхности. Каждое положение несёт бэклинк `realizes: US-N` к истории PRD, которую формализует (ADR-0014 §2).

**Авторинг мероприятия**

- **EARS-1** _(realizes: US-1)_ — Когда оператор отправляет форму создания мероприятия в админке, система должна создать мероприятие в **`draft`**, несущее название, школу/серию, **дату + время, введённые как МСК и хранимые как один каноничный момент**, длительность, описание, **спикеров как упорядоченный список свободно-текстовых записей (имя + строки регалий), валидируемых только как текст** (LD-1 — ссылки на реальные записи это волна 2, а поле остаётся расширяемым списком, чтобы вариант волны 2 был аддитивной миграцией), целевые специальности, **PDF программы, загруженный в object storage**, и ссылку на спонсора/партнёра; `draft`-мероприятие **не** публично достижимо (004 EARS-6), пока не опубликовано.
- **EARS-2** _(realizes: US-2)_ — Когда оператор редактирует мероприятие, система должна сохранить изменение в авторенный агрегат и отразить его на публичной странице мероприятия (004); в частности, **замена PDF программы после публикации** должна обновить ссылку на хранимый объект, так что публичная страница отдаёт **текущий** файл, а вытесненный файл **больше не отдаётся** — оператору никогда не нужно снимать с публикации, чтобы исправить деталь.
- **EARS-3** _(realizes: US-3)_ — Когда оператор/режиссёр конфигурирует стрим, система должна записать стрим-конфиг из **явного провайдера закрытого enum `rutube | youtube`** плюс **ссылки на embed** (provider-scoped id стрима, никогда не URL для сниффинга), исправимый пока мероприятие `published`; комната 006 должна инстанцировать плеер из **ровно этого конфига** и никогда не выводить провайдера из строки URL.

**Переходы жизненного цикла**

- **EARS-4** _(realizes: US-5)_ — Когда оператор публикует `draft`-мероприятие, система должна перевести его **`draft → published`**, сделав публично достижимым на странице + в листинге 004 и **открыв регистрацию** (005); публикация должна быть **отклонена, если мероприятие не в `draft`**, а переход должен добавить одну строку `audit_ledger`.
- **EARS-5** _(realizes: US-4)_ — Когда режиссёр **открывает комнату**, система должна перевести мероприятие **`published → live`** — впуская зарегистрированных врачей в комнату 006 и запуская фиксацию присутствия; а когда режиссёр **закрывает комнату**, система должна перевести его **`live → ended`** — останавливая допуск в комнату и приём heartbeat/чата и **ограничивая окно присутствия** (006 EARS-7). Открытие должно быть отклонено, если не `published`; закрытие — если не `live`; каждый переход добавляет одну строку `audit_ledger`.
- **EARS-6** _(realizes: US-5)_ — Когда оператор архивирует `ended`-мероприятие, система должна перевести его **`ended → archived`** как **ручное** действие (LD-2 — без планировщика, без автоматизации по времени в волне 1), после чего мероприятие **уходит со всех публичных поверхностей** (выпадение из листинга 004 + уведомление об архиве); архивирование должно быть **отклонено, если мероприятие не `ended`**, а переход должен добавить одну строку `audit_ledger`.
- **EARS-7** _(realizes: US-5)_ — Жизненный цикл мероприятия должен быть **единым полем `EventLifecycleState`** с **закрытым набором переходов** `draft→published`, `published→live`, `live→ended`, `ended→archived`; админ-UI должен **предлагать только переходы, валидные из текущего состояния**, а сервер должен **отклонять любой невалидный скачок** (например, `draft→live`, `published→ended`, повторное открытие `archived` или любой обратный ход) — принуждение **серверное**, не просто скрытие в UI, и **нет** перехода снятия с публикации / `published→draft` (PRD его не называет).

**Кросс-срезовое (повсеместное / нежелательное поведение)**

- **EARS-8** _(realizes: US-1, US-2, US-3, US-4, US-5)_ — Админ-команды (`CreateEvent`, `UpdateEvent`, `ConfigureStream`, `PublishEvent`, `OpenRoom`, `CloseRoom`, `ArchiveEvent`) и админ-read'ы (`EventAdminList`, `EventAdminDetail`) должны быть классифицированы **`access: authenticated`, `required_roles: platform_admin`, `auth_check: fast-path`** в матрице endpoint-authz (ADR-0001 §2) — одна доверенная админ-группа в волне 1 (LD-3), роли одной достаточно. **Ни один** переход жизненного цикла или write стрим-конфига/авторинга не должен быть вызываем `doctor_guest` или публичным вызывающим; админ-мутации идут по высокоставочному tier валидации (ADR-0001 §2.5/§8), а админ-приложение закрыто 2FA (ADR-0004). Производимые публичные проекции (`PublicEventPage`, `UpcomingBroadcastCard`) сохраняют `public`-классификацию 004.
- **EARS-9** _(realizes: US-1, US-2, US-4, US-5)_ — Система должна держать **один источник истины для состояния мероприятия**: `EventLifecycleState`, который пишут команды 007, — ровно то, что отображает админ-поверхность **и** что читают 004 (видимость листинга/страницы), 005 (гейтинг регистрации) и 006 (доступ в комнату) — не должно быть **второго** флага видимости и **никакой** булевой россыпи на сверку, так что админка и портал никогда не могут разойтись о состоянии мероприятия.
- **EARS-10** _(realizes: US-1, US-2, US-3)_ — Админ-поверхность 007 должна рендериться на **русском** с **отсутствием хардкод-строк** — весь копирайт (лейблы полей, имена состояний жизненного цикла, лейблы действий-переходов, сообщения валидации/ошибок, варианты провайдера стрима) берётся из типизированного каталога сообщений из 003 (EARS-21) и переиспользуемого в 004/005/006; любая **абсолютная** дата/время (дата/время эфира в списке и деталях) должны быть представлены в `Europe/Moscow` с пометкой **МСК**, вычислены из хранимого каноничного момента и никогда не съезжать в локальный пояс оператора.
- **EARS-11** _(realizes: US-1)_ — Админ-поверхность должна быть построена на **штатном UI Refine** (ADR-0004) с гейтом `build-ui-from-design-system` adopt-before-bespoke, прогоняемым первым; токен-дисциплина ADR-0013 (без произвольных значений Tailwind) должна применяться к любому bespoke-стайлингу. Поскольку **админ-канваса не существует** (зафиксированный Stage-A gap, PRD «Утверждённый мокап»), 007 не несёт **никакого** канвас-точного требования; выбор оставить штатный Refine vs заказать админ-канвас — **решение владельца на ближайшем чекпоинте Stage-A**, названное здесь, не решаемое 007.

## Инварианты

- Состояние мероприятия — **одно** поле `EventLifecycleState` с **закрытым** набором переходов; единственные легальные ходы — четыре форвардных перехода, любой невалидный скачок отклоняется на сервере, а админ-UI предлагает только текущевалидные переходы (EARS-7). Нет снятия с публикации / обратного хода.
- Состояние админки **есть** истина портала — состояние, которое пишет 007, — это состояние, которое читают 004/005/006; нет второго флага видимости, и они никогда не могут разойтись (EARS-9).
- Провайдер стрима всегда один из **закрытого enum** `rutube | youtube`, выбранный явно, с provider-scoped ссылкой на embed — никогда не URL для сниффинга (EARS-3).
- PDF программы живёт в **object storage** и **заменяем**; после замены публичная страница отдаёт текущий файл, а вытесненный файл больше не отдаётся (EARS-2).
- Любая абсолютная дата/время хранится как **один каноничный момент** и рендерится в `Europe/Moscow` с пометкой МСК, на админ-поверхности и в каждой производимой проекции (EARS-10).
- Каждая команда 007 и админ-read требует роли `platform_admin`; `doctor_guest` или публичный вызывающий **никогда** не допускается к авторингу, переходу или чтению админ-данных (EARS-8).
- Каждый переход жизненного цикла добавляет **ровно одну** терминальную строку `audit_ledger` (EARS-4, EARS-5, EARS-6; ADR-0003 §6).
- `ended → archived` — **ручное** действие оператора — в волне 1 его не запускает планировщик (EARS-6, LD-2).

## Зависимости и секвенирование

- **007 — производитель, закрывающий швы волны 1.** 004 (страница/листинг мероприятия), 005 (регистрация) и 006 (комната) строились + E2E-прогонялись на **seed-мероприятиях + seed-стрим-конфиге + seed-списке**, потому что 007 не приземлилась; 007 приземляет **реальный авторенный агрегат + жизненный цикл + стрим-конфиг**. Критерий «сделано против реальной зависимости» для этих швов — _«вертикали 004/005/006 работают против мероприятий, авторенных, переведённых и стрим-сконфигурированных через админ-приложение 007, а не seed'ов»_ — несомый на блокирующих связках 004↔007 / 005↔007 / 006↔007 (`open-ears-issues` шаг 4). 007 — производящая сторона ровно тех швов, которые 005/006 назвали с потребляющей стороны.
- **MVP-критичный дедлайн — 2026-07-17.** Минимальный путь создать → опубликовать → сконфигурировать стрим → открыть → закрыть должен быть проходим одним оператором/режиссёром для живого вебинара, без вмешательства разработчика (приёмка PRD). Happy-path E2E 007 — эта дуга в админ-приложении.
- **Фундамент auth (фича 003) — отгружен.** Админ-принципал — сессия `platform_admin` из отгруженного слоя 003/IdP; 007 не добавляет auth-примитивов (EARS-8). Админ-приложение Refine закрыто 2FA (ADR-0004 §3).
- **Object storage (Timeweb Object Storage / MinIO дев-стенда).** PDF программы загружается в object storage; e2e-тест загрузки гоняется против MinIO дев-стенда (`S3_ENDPOINT`), `skipIf` при отсутствии — эндпоинты/бакет читаются из `.env.local`, никогда не хардкодятся (`.claude/rules/dev-stand.md`; ADR-0003).
- **Публичная страница + листинг (фича 004).** 004 рендерит производимые `PublicEventPage`/`UpcomingBroadcastCard`; 007 авторит данные за ними и владеет переходами жизненного цикла, которые 004 читает (`EventPublished`/`WentLive`/`Ended`/`Archived`). Публикация/архивирование через 007 — это то, что переключает публичную достижимость 004 с seed'ов на реальный агрегат.
- **Регистрация (фича 005).** 005 гейтирует affordance регистрации на `EventLifecycleState`, который пишет 007 (published/live предлагается; ended/archived скрывается). Публикация 007 открывает регистрацию; 007 не владеет записью регистрации.
- **Вебинарная комната (фича 006).** 006 читает стрим-конфиг 007 (enum провайдера + ссылка на embed), чтобы инстанцировать плеер, и гейтирует допуск на `live`; **открытие/закрытие 006, названные её швом 007, — это команды 007 `OpenRoom`/`CloseRoom`** (EARS-5). Критерий «сделано против реальной зависимости» 006 — комната открывается/закрывается через режиссёрские органы 007 и инстанцирует из авторенного 007 конфига — удовлетворяется этой фичей.
- **Админ-фреймворк (Refine, ADR-0004).** Админ-поверхность — штатный Refine + кастомные data/auth/access-провайдеры над NestJS API; админ-канваса не существует (Stage-A gap), поэтому зависимости от канваса дизайн-системы нет. Гейт `build-ui-from-design-system` adopt-before-bespoke прогоняется для любого bespoke-элемента; токен-дисциплина ADR-0013 применяется там, где появляется кастомный стайлинг (EARS-11).
- **Dev-стенд Postgres + Zitadel + MinIO.** Админ-e2e-тесты гоняются против Postgres дев-стенда (`DATABASE_URL`) + `idp` живого стенда (`IDP_ISSUER`, для сессии `platform_admin`) + MinIO (`S3_ENDPOINT`) по `.claude/rules/dev-stand.md`; эндпоинты/порты читаются из `.env.local`.
