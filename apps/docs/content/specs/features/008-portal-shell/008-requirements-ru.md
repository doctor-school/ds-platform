# 008 — Оболочка портала и discovery-фасад (Требования)

Написано из PRD [`008-product-ru.md`](./008-product-ru.md) (ADR-0014). Каждый пункт EARS несёт бэклинк `realizes: US-N` на продуктовую историю, которую он формализует. US-6 в PRD — _Ретайрен_ (выход живёт на профиле, фича 009) — EARS для него не пишется; пропуск намеренный.

## Outcomes (Результаты)

- **Постоянный хедер app-shell** присутствует на каждом маршруте портала и несёт **логотип** (→ `/`), верхнюю навигацию **[Эфиры · Мои события]**, **переключатель темы** и — для залогиненного врача — **иконку-аватар с инициалами, которая ведёт прямо в профиль `/account`** (иконка, а **не** дропдаун-меню); гость вместо неё видит кнопку **«Войти»**. В хедере **нет «Выйти»** (выход живёт на профиле, фича 009).
- **`/` — канонический публичный discovery-фасад** — листинг ближайших эфиров (переиспользует поверхность листинга фичи 004, завендоренный `webinars-listing.dc.html`), рендерящийся **одинаково** гостю и залогиненному врачу. Discovery никогда не ветвится по состоянию авторизации; ветвится только аффорданс аккаунта в хедере. Отдельного дашборда нет.
- **Посадка после логина — `/`.** Auth-флоу (фича 003) возвращает врача на discovery-фасад, никогда на каркас.
- Навигация хедера разрешается только в **отгруженные** поверхности: «Эфиры» → `/` (этот discovery-фасад), «Мои события» → `/account/events` (фича 005), иконка-аватар → `/account` (фича 009, профиль).
- **Каркас `/` ретайрен** — заглушечная карточка «Каркас приложения» (единственное действие которой — кнопка «перейти ко входу») больше не достижима в портале.
- Хедер и `/` **уже спроектированы** в завендоренном канвасе «Doctor.School визуальный язык» ([`design-source/`](../../../../../../design-source/README.md)); эта фича **монтирует существующую оболочку**, не изобретает её, и строится по файлам канваса (не по prose из Issue) согласно ADR-0013.

## Scope

**В scope:**

- **Постоянный хедер app-shell**, смонтированный на каждом маршруте портала: логотип (→ `/`), верхняя навигация **[Эфиры · Мои события]**, переключатель темы и аффорданс аккаунта по состоянию авторизации (иконка-аватар→`/account` для залогиненного врача / **«Войти»** для гостя).
- **Разрешение маршрутов навигации** в отгруженные поверхности (логотип и «Эфиры» → `/`; «Мои события» → `/account/events`, фича 005; иконка-аватар → `/account`, фича 009).
- **Переключатель темы** (светлая/тёмная, с сохранением — завендоренный канвас использует `localStorage['ds-theme']`).
- **Ветвление по состоянию авторизации** в хедере: залогинен → иконка-аватар (инициалы, не дропдаун, без «Выйти»); гость → «Войти». Хедер правдиво отражает состояние сессии, читаемое из `GET /v1/auth/session` (фича 003).
- **Посадка после логина = `/`** — auth-флоу фичи 003 возвращает авторизованного врача на discovery-фасад.
- **`/` как публичный discovery-листинг**, переиспользующий поверхность листинга фичи 004 (завендоренный `webinars-listing.dc.html`), рендерящийся одинаково гостю и врачу.
- **Ретайр каркаса `/`** — заглушечная карточка «Каркас приложения» удалена и больше не достижима.
- **Мобильное сворачивание** — верхняя навигация сворачивается в `≡` дропдаун с теми же [Эфиры · Мои события].
- **RU-primary копия хедера** из message-каталога (без хардкода user-facing строк), согласованно с EARS-21 фичи 003.

**Явно вне scope** (каждый — задокументированная отсрочка, не молчаливый стаб):

- **«Школы».** «Школы» **не рендерится в навигации v1** (решение владельца 2026-07-15). У неё пока нет ни фичи, ни канвасного пункта назначения; вместо отгрузки видимо-инертного пункта навигация опускает её целиком, пока не появится её собственная фича. «Школы» возвращается в навигацию через discovery этой фичи, а не как заглушка здесь. (PRD → Вне scope.)
- **Внутренности discovery-листинга** (карточки, порядок, сигналы жизненного цикла, empty/error-состояния самого листинга) — принадлежат **фиче 004**; `/` переиспользует эту поверхность, не переспецифицирует её.
- **Содержимое «Мои события» и комната эфира** — фичи **005 / 006**. Эта фича лишь монтирует цель навигации.
- **Содержимое профиля и выход** — профиль на `/account` **уже отгружен** (фича 009 суперседнута [#770](https://github.com/doctor-school/ds-platform/issues/770); `/account` — реальный профиль врача, не debug-дамп). Иконка-аватар лишь навигирует туда; экран профиля эта фича не строит.
- **Поиск, фасеты, уведомления, более богатый дом ЛК** — будущие итерации (бриф эпика).

## Constraints (Ограничения)

- **UI из дизайн-системы (ADR-0013).** Хедер и `/` строятся из `@ds/design-system`, стилизация **только** через токены (произвольные значения Tailwind заблокированы линтером). Каждый канвас, который рендерит поверхность, завендорен под [`design-source/`](../../../../../../design-source/README.md) и строится по файлам, не по prose. Постоянный хедер — это хедер завендоренного канваса, присутствующий на `webinars-listing.dc.html`, `my-events.dc.html` и `webinar-page.dc.html`; где prose и канвас расходятся — канвас выигрывает.
- **Никакого хардкода origin / маршруты-как-конфиг.** Цели маршрутов (`/`, `/account`, `/account/events`, маршрут логина) разрешаются через слой роутинга портала, не дублируются строками по компонентам. Origin портала читается из конфигурации (зеркалит Constraints фичи 003).
- **Discovery не ветвится по авторизации.** `/` рендерит одну поверхность и гостю, и врачу; единственный зависящий от авторизации элемент — аффорданс аккаунта в хедере (EARS-4 / EARS-5). Вариант `/` по состоянию авторизации запрещён.
- **Хедер не владеет выходом и дропдауном.** Аффорданс аккаунта — иконка-ссылка на `/account`, никогда не меню; «Выйти» — аффорданс профиля фичи 009. Это продуктовое решение (приёмка PRD), не вольность реализации.
- **Сессия как read-only вход.** Эта фича потребляет состояние сессии фичи 003 (`GET /v1/auth/session`), чтобы выбрать ветку хедера; она **не вводит нового бэкенд-примитива авторизации** и не минтит сессию (посадка после логина — целевой redirect, переданный существующему флоу фичи 003).
- **Стек (ADR-0004).** `apps/portal` — Next.js 15 + Refine; оболочка компонуется как layout уровня приложения, чтобы хедер присутствовал на каждом маршруте по построению.

## Prior decisions (Прежние решения)

- **ADR-0014** Двухуровневая продуктовая спека — PRD [`008-product-ru.md`](./008-product-ru.md) — источник; каждый EARS несёт `realizes: US-N` (§2). Канвас — _источник_, репозиторий держит _собранный артефакт_ (§4). Прошлый арт (legacy ЛК) майнится read-only через бриф эпика (§3), никогда как целевой layout.
- **ADR-0013** Дизайн-система и canvas-derived UI — только токены, adopt-before-bespoke, вендорить каждый канвас, который рендерит поверхность, поэлементный паритет рендера на обоих брейкпоинтах × обеих темах (Stage-B).
- **ADR-0004** Фронтенд-приложения — портал — `apps/portal` (Next.js 15 + Refine); оболочка монтируется как app-shell layout.
- **ADR-0006 §4** Документация и SSOT — структура триплета фича-спеки + плоская нумерация EARS.

## Event Model (Модель событий)

Это **фича UI-композиции / ИА**, не новый бэкенд-агрегат. Она не вводит команду, мутирующую серверное состояние, и не владеет новой персистируемой read-моделью; она компонует отгруженные поверхности и читает сессию фичи 003. Модель ниже выражена в терминах фронтенда.

### Команды (клиентские навигации / UI-интенты — без серверной мутации)

`RenderShell` (смонтировать постоянный хедер на маршруте) · `NavigateNav` (разрешить пункт верхней навигации в его отгруженный маршрут) · `ToggleTheme` · `OpenProfile` (иконка-аватар → `/account`) · `GoToLogin` (гостевой «Войти») · `LandAfterLogin` (auth-флоу фичи 003 возвращает на `/`).

### События

| Событие                  | Владелец                | Заметки                                                       |
| ------------------------ | ----------------------- | ------------------------------------------------------------- |
| `ShellRendered`          | `apps/portal`           | Хедер app-shell присутствует на текущем маршруте.             |
| `ThemePreferenceChanged` | `apps/portal`           | Сохраняется в `localStorage['ds-theme']` (конвенция канваса). |
| `PostLoginLanded`        | фича 003 → эта оболочка | Auth-флоу приземляет врача на `/`.                            |

### Read-модели

- **`AuthState`** — производится из `GET /v1/auth/session` (фича 003): `{ authenticated, initials? }`. Управляет ветвлением аффорданса аккаунта в хедере (иконка-аватар vs «Войти»); инициалы врача рендерят аватар. Ничего здесь эта фича не персистирует.
- **`DiscoveryListing`** — поверхность листинга ближайших эфиров фичи 004, переиспользуемая дословно на `/`; принадлежит фиче 004, потребляется здесь.

### Политики

- **На рендер любого маршрута портала** → хедер app-shell смонтирован (`ShellRendered`) — хедер инвариант layout, не per-page opt-in.
- **На успешный логин фичи 003** → приземлить врача на `/` (`PostLoginLanded`).
- **На `AuthState.authenticated` true/false** → рендерить аффорданс иконка-аватар→`/account` vs кнопку «Войти»; никогда не вариант контента `/` по авторизации.

## EARS-требования

> **Конвенция нумерации:** плоская (`EARS-1`, `EARS-2`, …) по ADR-0006 §4. Каждый пункт несёт бэклинк `realizes: US-N` на PRD (ADR-0014 §2). EARS-1…9 + EARS-11 — функциональные хендлеры (кандидаты в дочерние Issue); EARS-10 — _Ретайрен_ (см. ниже); EARS-12…13 — сквозные ubiquitous-требования, применяемые ко всей поверхности.

**Постоянный хедер app-shell**

- **EARS-1** (realizes **US-1**): На каждом маршруте портала система должна рендерить постоянный хедер app-shell, несущий логотип (ссылка на `/`), верхнюю навигацию **[Эфиры · Мои события]** и переключатель темы — чтобы хедер присутствовал с любой страницы, а не только по прямой ссылке. Хедер компонуется из завендоренного канваса через `@ds/design-system` (Constraints).
- **EARS-2** (realizes **US-1**): Когда пользователь активирует цель навигации хедера, система должна разрешить её в её **отгруженную** поверхность: логотип и «Эфиры» → `/` (discovery-фасад), «Мои события» → `/account/events` (фича 005). Каждая цель навигации разрешается в отгруженную поверхность — нет ни отложенной, ни инертной цели.
- **EARS-3** (realizes **US-1**): Когда пользователь активирует переключатель темы, система должна переключить портал между светлой и тёмной и сохранить предпочтение (завендоренный канвас использует `localStorage['ds-theme']`), чтобы выбор пережил навигацию и перезагрузку.

**Аффорданс аккаунта по состоянию авторизации**

- **EARS-4** (realizes **US-2**): Пока вызывающий — гость (нет авторизованной сессии по `GET /v1/auth/session`, фича 003), система должна рендерить в хедере кнопку **«Войти»**, ведущую на поверхность логина — чтобы у гостя всегда был очевидный вход. Хедер не показывает ни аватара, ни «Выйти».
- **EARS-5** (realizes **US-5**): Пока вызывающий — залогиненный врач, система должна рендерить в хедере **иконку-аватар с инициалами врача** — иконку, а **не** дропдаун-меню — и **не** рендерить «Выйти» в хедере (выход живёт на профиле, фича 009).
- **EARS-6** (realizes **US-5**): Когда залогиненный врач активирует иконку-аватар, система должна навигировать на профиль `/account` (фича 009) — один тап, один пункт назначения, без промежуточного меню.

**Discovery-фасад и посадка после логина**

- **EARS-7** (realizes **US-3**): Когда врач завершает логин (фича 003), система должна вернуть его на **`/`** — листинг ближайших эфиров — как посадку после логина, никогда на заглушечную карточку или мёртвый дашборд.
- **EARS-8** (realizes **US-4**): Система должна рендерить `/` как **публичный** discovery-листинг ближайших эфиров, переиспользуя поверхность листинга фичи 004 (завендоренный `webinars-listing.dc.html`), **одинаково** гостю и залогиненному врачу; `/` не должен ветвить контент по состоянию авторизации — различается только аффорданс аккаунта в хедере (EARS-4 / EARS-5).
- **EARS-9** (realizes **US-3**): Система должна ретайрить каркас `/` «Каркас приложения» — заглушечная карточка, единственное действие которой — кнопка «перейти ко входу», больше не должна быть достижима в портале, а `/` вместо неё обслуживает discovery-листинг (EARS-8).

**Отложенная цель навигации и мобильный**

- **EARS-10** — _Ретайрен._ «Школы» убрана из навигации (владелец 2026-07-15); инертная цель не рендерится. Навигация v1 — [Эфиры · Мои события], и каждая цель разрешается в отгруженную поверхность (EARS-2). Id сохранён для плоской нумерации (репо-паттерн, ср. US-6); хендлер не пишется. «Школы» возвращается через discovery своей собственной фичи, когда та появится (Scope → Вне).
- **EARS-11** (realizes **US-1**): Пока вьюпорт на мобильном брейкпоинте (канвасный `≤900px`), система должна сворачивать верхнюю навигацию в `≡` дропдаун с теми же **[Эфиры · Мои события]**, сохраняя разрешение каждой цели навигации (EARS-2).

**Сквозные (ubiquitous)**

- **EARS-12** (realizes **US-1**): Система должна строить постоянный хедер и `/` из завендоренного канваса «Doctor.School визуальный язык» через токены `@ds/design-system` (ADR-0013) — без bespoke-элемента без гейта registry-research, без произвольных значений Tailwind — и отрендеренный результат должен совпадать с канвасом поэлементно на **обоих брейкпоинтах × обеих темах** (проверка паритета рендера Stage-B).
- **EARS-13** (realizes **US-1**): Хедер не должен нести **хардкоженых user-facing строк** — метки навигации («Эфиры», «Мои события»), кнопка «Войти» и доступные метки переключателя темы / аватара берутся из RU-primary message-каталога (согласованно с EARS-21 фичи 003), чтобы будущая локаль была аддитивной без переработки компонентов оболочки.

## Invariants (Инварианты)

- Хедер app-shell присутствует на **каждом** маршруте портала (EARS-1) — маршрут без него — дефект.
- `/` рендерит **одну** поверхность независимо от состояния авторизации (EARS-8); единственный зависящий от авторизации элемент хедера — аффорданс аккаунта (EARS-4 / EARS-5). Варианта контента `/` по авторизации не существует.
- Хедер не содержит **аффорданса выхода и дропдаун-меню** (EARS-5) — аффорданс аккаунта — только иконка-ссылка на `/account`.
- Каждая цель навигации хедера разрешается в **отгруженную** поверхность (EARS-2) — ни инертной, ни отложенной цели навигации не существует.
- Каркас `/` «Каркас приложения» **недостижим** после этой фичи (EARS-9).
- Эта фича не минтит сессию и не пишет нового персистируемого состояния; она читает сессию фичи 003 как единственный вход авторизации (Constraints).

## Dependencies & sequencing (Зависимости и последовательность)

- **Зависит от отгруженных поверхностей — все уже на `main`.** `/` переиспользует поверхность листинга **фичи 004** (завендоренный `webinars-listing.dc.html`); «Мои события» → **фича 005** (`/account/events`); иконка-аватар → **профиль `/account`** (отгружен через [#770](https://github.com/doctor-school/ds-platform/issues/770), суперседит фичу 009). Фичи 004/005 и профиль `/account` — **все отгружены**: каждая цель аватара/навигации, которую монтирует эта оболочка, разрешается в уже существующую поверхность, так что у 008 **нет несобранной зависимости**. Это чистый слайс UI-композиции над отгруженными поверхностями; хедер не владеет ни одной из них.
- **Источник дизайна (канвас) завендорен.** Канвасы хедера + `/` (`webinars-listing.dc.html`, `my-events.dc.html`, `webinar-page.dc.html`) уже под `design-source/` (ADR-0013). Никакой pull DesignSync не является предпосылкой авторинга; паритет рендера Stage-B (EARS-12) проверяется против них.
- **Stage-B live-verify.** Как `user-facing`-поверхность, брендированный рендер переподтверждается продуктовым владельцем на запущенном стенде перед merge (или под назначенным батч-гейтом Stage-B эпика, если он объявлен). Стенд остаётся поднятым до этого вердикта.
- **«Школы» — не seam.** Навигация v1 опускает «Школы» целиком (EARS-10 _Ретайрен_) — инертной заглушки нет, значит и seam-декизион-дебта трекать нечего. «Школы» становится целью навигации только когда её собственная фича будет заспецирована и построена, через discovery этой фичи.
